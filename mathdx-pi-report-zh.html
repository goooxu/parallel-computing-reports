1. 开篇：这篇教程教什么

在 GPU 上做数学，有两种截然不同的用库姿势。第一种是打电话：调用 cuFFT、cuBLAS 这类主机端库，一个函数进去，库替你把事情做完——它自己发射 kernel、自己管调度，你等结果。第二种是把库请进自己的 kernel：cuFFTDx 和 cuBLASDx（NVIDIA MathDx 里的两个 header-only 模板库）不替你做完任何事，它们只是把「一个线程块内的小 FFT / 小矩阵乘」变成你 CUDA 代码里的一行普通语句，数据在你自己的寄存器和共享内存里，进出这行语句前后的每一个字节搬运都归你管。

它们是谁
Dx = Device Extensions（设备端扩展）。二者同属 NVIDIA MathDx 合集——与 CUDA Toolkit 相互独立、单独发布的纯头文件 C++ 模板库，家族里还有 cuSolverDx（分解/特征值）、cuRANDDx（随机数）、nvCOMPDx（压缩）。cuBLASDx 构建在开源模板库 CUTLASS 之上。
年表与要求
cuFFTDx 最年长：经历早期体验阶段后于 2022 年发布 1.0 正式版（MathDx 合集的版本号从 22.02 起算，格式为「年.月」）；cuBLASDx 年轻得多——从 0.1 早期版起步，2024 年还在 0.2，至今未到 1.0，API 仍在快速演进。较新的 MathDx 要求 CUDA 13.0+（CUDA 11/12 支持已移除），架构覆盖 Ada/Hopper/Blackwell 等。

这篇教程完整走一遍第二种姿势：怎么配置、怎么在一个 kernel 里组合两个库、怎么把「一个块装得下」的小变换拼成 228 点的大 FFT，以及——最重要的——「自己掌管数据进出」这件事到底值多少性能。为了让答案有分量，我们选了一个足够狠的贯穿案例：在一块 H100 上，把 π 计算到 100,000,000 位，全部大数运算从零实现。先亮成绩单：

成绩单之外，这篇教程真正想讲的是「怎么到达这里」：后半部分逐项拆解这条流水线的性能工程——原子归约、省字节、批量化、访存形态，以及压轴的 kernel 深度融合。其中最深的那一层改造（把纯搬运工作折进计算 kernel 的装载/写回相位），只有当 FFT 是「你 kernel 里的一行」时才写得出来——这正是设备端库的独门优势，也是本文标题的含义。

2. 并行第一课：从「循环」到「整片数组」

CPU＝几位博士生
核心少但强，擅长「一步依赖上一步」的复杂逻辑；跑 for 循环是一次一格。
GPU＝上万名小学生
这块 H100 有一万多个 CUDA 核心，每个只会简单算术，但能同时开工。

GPU 程序的最小组织单位是线程；线程按线程块分组（本文的块多为 256 线程），同一块内的线程可以用一小片超快的共享内存（每块最多 227KB，通常用几十 KB）协作，块与块之间只能通过显存见面。要让上万个核心都有活干，就得把「逐元素的循环」改写成「对整片数组的一次运算」：

全文最重要的一句话：后面每一个大数运算，我们都会把它写成「对整片 limb 数组的运算」，这样它才真正跑在 GPU 上。做不到这一点的运算（比如天生串行的逐位进位），就要找到它的并行等价形式——这是第 5 节和第 12 节的主题。

这一课还有下半句，也是本文的独特视角：把循环改成整片运算之后，每一遍「整片」都是一次全量显存读写。一亿位 π 的中间数组动辄 4GB，多走一遍显存就是多花几毫秒。谁掌管这些「遍数」，谁就掌管性能——这正是下一节两种用库模型的分水岭。

3. 打电话，还是请进厨房

先看熟悉的主机调用模型。用 cuFFT 做一次「变换—逐点乘—逆变换」，主机代码的典型形态是：

// 主机调用模型的典型形态（示意）：每一步都是一次「打电话」
cufftHandle plan;
cufftPlan1d(&plan, N, CUFFT_Z2Z, 1);            // ① CPU 上先建 plan（句柄 + 库内部状态机）
my_pack_kernel<<<grid, 256>>>(limbs, buf);      // ② 前处理得自己先发射一个 kernel
cufftExecZ2Z(plan, buf, buf, CUFFT_FORWARD);    // ③ 打电话：库发射它自己的 kernel 完成 FFT
my_pointwise_kernel<<<grid, 256>>>(buf, spec);  // ④ 后处理又是一个自己的 kernel
cufftExecZ2Z(plan, buf, buf, CUFFT_INVERSE);    // ⑤ 逆变换（再打一次电话）
// 每步之间数据都要整片经过显存；库 kernel 内部一行也改不了

这个模型的合同很清楚：你在 CPU 上建好 plan（库的执行计划句柄），把数据放进显存，打一个电话，库发射它自己的 kernel 把这一步做完。省心，成熟，快——但有两条写死在合同里的限制：①每一步之间，数据必须整片躺回显存（库 kernel 的输入输出只认显存指针）；②库 kernel 内部你一行代码都插不进去（想在装载时顺手做个类型转换？对不起，先自己发射一个 kernel 把类型转好）。

设备内联模型把这两条合同全部撕掉。cuFFTDx/cuBLASDx 是纯头文件的 C++ 模板库：你在编译期用「+ 号拼配料」的方式声明想要什么变换，它给你一个类型；在你自己的 kernel 里，这个类型的 execute() 就地完成一个线程块内的 FFT 或矩阵乘——不发射新 kernel，数据全程在你的寄存器和共享内存里：

// 设备内联的最小骨架（由 cuda_agm/fft.cuh 的 k_stageD 简化）——库是你 kernel 里的一行
template<unsigned Q> __global__ void my_fft_rows(cplx* data) {
    using FFT = decltype(cufftdx::Block() + cufftdx::Size<Q>() + /* 菜谱见下文 */ ...);
    extern __shared__ char smem[];              // 库要的共享内存工作区，你来给
    dxcplx rd[FFT::storage_size];               // 数据住在你的寄存器里
    for (i...) rd[i] = data[offset + e];        // 装载相位：你写的循环，想干什么都行
    FFT().execute(rd, smem);                    // ← 库在这儿：块内 FFT，不发射新 kernel
    for (i...) data[offset + e] = rd[i];        // 写回相位：也是你写的循环
}
主机调用（cuFFT/cuBLAS）设备内联（cuFFTDx/cuBLASDx）
谁发射 kernel库（黑盒）你（库只是 kernel 里的一行）
数据在哪每步进出显存你的寄存器 / 共享内存
能否融合前后处理基本不能（唯一的官方口子是回调机制，窄且脆）随便融——装载/写回相位是你的代码
规模上限近乎任意一个线程块装得下的规模（fp64 几千点）
成熟度二十年打磨你自己拼，正确性自己保证（第 16 节）
全文的伏笔：FFT().execute(rd, smem) 这行代码前后的每一个字节搬运都归你管。第 12–15 节性能工程里最深的一层，就长在「装载相位顺手做 twiddle」「写回相位顺手做取整」这些主机调用模型里根本不存在的动词上。

当然，天下没有免费的午餐：设备内联模型的上限是「一个线程块装得下」，而我们需要 228 点——大 FFT 得自己拆、自己拼（第 11 节）；拼错了没人兜底，所以还得自己搭一副验证梯子（第 16 节）。先看看这个案例到底有多大。

4. 运行示例登场：一亿位 π，为什么它能逼出一切

为什么不能直接算？浮点数装不下。double 只有约 15～16 位有效数字，离一亿位差了 7 个数量级——这不是精度不够，而是类型里根本没那么多位置。而 GPU 上又没有现成的「大整数」类型，所以要自己用一大片小数字拼出一个大数（第 5 节），再把大数的加减乘除全部翻译成整片数组运算。这个案例的负载画像如下：

limb 数组长度33,333,350每 limb 3 位十进制（基 1000，int32）

为什么选它当教程案例？因为它把设备端工作流的每一个环节都逼到了台前：大 FFT 必须自己从块级积木拼起（228 点远超一个块的容量）；1062 次大乘法让每一遍多余的显存读写都被放大一千倍；26 轮迭代、几百步牛顿让 kernel 发射开销无处藏身；而「必须逐位正确」让所有 trick 都要过验证这一关。

角色分工：π 的数学（第 5–9 节）是积木——它们决定我们要算什么；设备端方法论（第 10–15 节）是主角——它决定同样的数学算得多快。赶时间的读者可以把第 5–9 节当参考手册跳过，不影响主线。

5. 数学积木 I：limb 大数、进位，与 999…9 暗礁

表示。把大整数拆成许多 0～999 的小数字，每个叫一个 limb，整体是一个 int32 数组（低位在前）。选基 B=1000（=10³）是因为最终要输出十进制：每个 limb 恰好 3 位数字，输出零成本。一亿位的 π 就是约 3,333 万个 limb 的数组（130MB 出头）。

加法与进位。小学的「逐位相加、逢千进一」是天生串行的（第 k 位的进位要等第 k−1 位），正撞在第 2 节心法的枪口上。GPU 上改成两步：先整片相加（有些 limb 暂时超过 999，不管它），再并行传播进位。传播这一步藏着全项目最险的暗礁：牛顿迭代（第 7 节）会系统性地制造1 后面跟几十万个 0（或差一点时的 999…9）这种数——一个 +1 的进位要穿过整条 0/9 长链。「多趟传播、每趟推一格」的朴素并行版在这种输入上退化成 O(n) 趟：项目早期（百万位规模试验时）实测过一次乘法的进位传播了 166,679 趟、耗时 34.9 秒。

解法与硬件加法器的经典答案同源——进位前瞻（carry-lookahead）。观察：残余进位只会「穿过」999（借位只会穿过 000），在其他任何数字上都会被吸收。于是「位置 i 会不会收到进位」＝「i 下方最近的非 999 位置是否点火」——这是一个前缀扫描（prefix scan），一次算完所有位置，链再长也一趟到底：

# 串行进位（一次只能走一格）              # 并行前缀版（一次扫描走到底）
carry = 0                                 # 残余进位 ∈ {0,±1} 时：
for i in range(n):                        #   999 会让进位「穿过」(propagate)
    v = s[i] + carry                      #   ≥1000 必然「产生」进位 (generate)
    s[i] = v % 1000                       # (g,p) 是可结合的半群 → 分块前缀扫描
    carry = v // 1000                     # CUDA 三个 kernel：块内聚合→块间串链→应用
教训：向量化原语的「平均情况」性能可能藏着 O(n) 的最坏情况，而数值算法恰恰会系统性地制造最坏情况输入。快算法会把压力转移给基础设施。

减法用对称的借位写法；比较大小从最高位找第一个不同的 limb；「除以 2」有个漂亮的并行技巧：基 1000 是偶数，第 i 位结果 = d[i]/2 + 500·(d[i+1] 是否为奇)，所有 limb 一次算完、无需传播。这些原语的 CUDA 实现全在 cuda_agm/common.cuh。进位的最终形态（两阶段精确进位）在第 14 节。

6. 数学积木 II：FFT 乘法与 0.25 红线

把大数看成「1000 进制多项式」，两数相乘＝系数做卷积。朴素卷积是 O(n²)——对 3,333 万 limb 大约是 1015 次乘加，即使 H100 也要算上几十分钟，而我们要做 1062 次。卷积定理给出近路：

x⋅y=F−1(F(x)⊙F(y)), ☉ 为逐点相乘

浮点怎么算出精确整数？FFT 内部用 float64，但 float64 能精确表示到 253 ≈ 9×1015。一亿位时卷积系数最大约 n·999² ≈ 3.34×107×106 ≈ 3.3×1013——离上限还有约 270 倍。但这只是「系数本身装得下」；FFT 的几千万次浮点乘加还会累积舍入误差，所以每次乘法都设一道安全阀：检查每个浮点结果距最近整数的距离，超过 0.25 立即报错退出——绝不悄悄算错：

// 节选自 cuda_agm/fft.cuh 的 mul_fft_path —— 每次乘法的 0.25 红线
k_rint_err / k_perm_rint ...;                        // 取整 + |浮点−整数| 的块内最大归约
CUDA_CHECK(cudaMemcpy(&ebits, g_err, 8, cudaMemcpyDeviceToHost));
double err = __longlong_as_double(ebits);
if (err > g_fft_err_max) g_fft_err_max = err;        // 全程记录最大值（写进统计 JSON）
if (err >= 0.25) {                                   // 距最近整数超过 1/4 → 立即报错退出
    fprintf(stderr, "FFT rounding error too large: %g (m=%ld N=%ld)\n", err, m, N);
    std::exit(1);                                    // 绝不悄悄算错
}

一亿位全程的实测最大误差是 0.0176（Dx-GEMM 路径）/ 0.00879（纯 cuFFTDx 路径）——比红线低一个数量级，但已经能看出：基 1000 在 228 点这个规模上接近极限了（再大十倍就得换基 100）。红线的价值正在于此：它把「精度快不够了」变成显式可观测的量，而不是某天悄悄输出一串错误数字。

很小的乘法（两数长度之积——即 O(n²) 的乘加次数——≤ 1.6×107）直接用 schoolbook kernel，躲开 FFT 的固定开销——牛顿爬升的低层级全是这种小乘法。

7. 数学积木 III：牛顿迭代——只用乘法造出除法与开方

算 π 还需要除法和开方，但我们只想用已经造好的乘法。牛顿迭代做到了。求倒数 1/T：

xn+1=xn⁢(2−T⋅xn)

求倒数平方根 1/√K（开方即 √K = K·(1/√K)）：

yn+1=yn⁢(3−K⋅yn2)2

两个迭代都只有乘法、减法和「除以 2」（第 5 节的并行取半），而且都二次收敛——每迭代一步，正确位数翻倍：15 → 30 → 60 → … → 108，从一个 double 种子出发约 23 步到位。在整数世界里，1/√K 被表示成 y ≈ Bh/√K 的定点放大形式（h 比 K 的 limb 数略大，实现中取 h = len(K)+2；于是 K·y² ≈ B2h，第 9 节代码里的残差 B2h − K·y² 正来源于此），「除以 B 的幂」恰好是把数组低位整段丢掉——精确且零成本。

记住「每步位数翻倍」——第 8 节选算法、第 9 节的两件武器，全部建立在这一个性质上。

8. 算法主角：Gauss–Legendre 与定点缩放

基础设施齐了，用什么公式算 π？最出名的莱布尼茨级数是反面教材：

π4=1−13+15−17+⋯

一百万项才换来 6～7 位。历史上有收敛快得多的级数（好的每项贡献十几位），但级数有个共同天花板：正确位数与项数成正比（线性收敛），而且海量小项的合并是 GPU 最不喜欢的碎负载。Gauss–Legendre（Brent–Salamin，1976）不做级数求和，而是反复施加一组极简单的变换：

a0=1,b0=12,t0=14,p0=1
an+1=an+bn2, bn+1=anbn, tn+1=tn−pn(an−an+1)2, pn+1=2pn
π≈(an+bn)24tn

a 取算术平均、b 取几何平均，两者以惊人的速度彼此靠近（共同极限叫「算术几何平均数」AGM）。关键性质与牛顿同款：二次收敛，一致位数每轮翻倍——一亿位只要 26 轮，且每轮都是整片数组级别的大运算，正是 GPU 喜欢的负载形状：

代价同样明确：每一轮都要在一亿位精度上开一次平方根（b′=√(ab)），收尾还有一次同精度除法。这个算法能不能快，完全取决于大数乘法有多快、牛顿迭代做得聪明不聪明——正好把压力全部传导给我们的基础设施。

定点缩放解决「实数进整数库」：所有量一律乘 10p 存成整数（p = 100,000,050，3 的倍数 → 恰好 33,333,350 个 limb）。每个实数运算都严格对应一个整数运算：

实数世界定点整数世界（一切 ×10p）用到的原语
(a+b)/2(A+B)/2加法 + 偶基取半（无进位传播）
√(a·b)isqrt(A·B)：A·B 带 102p，开方后恰好回到 10p乘法 + 牛顿开方
t − p·(a−a′)²T − pₙ·(D² 右移 p 位)乘法 + 移位（丢低位=截断）
(a+b)²/(4t)(A+B)²·recip(4T) 再右移乘法 + 牛顿倒数

截断误差每轮只污染最低几位，26 轮累计远小于 50 位保护位（先在小规模上逐位验证过再上一亿）。主循环的骨架：

// 节选自 cuda_agm/pi_agm.cu 的 run_pi —— AGM 主循环（4 个环形槽位复用显存）
while (true) {
    g_arena.restore(mark0);                          // 每轮开始：临时量全部回收
    int c = big_cmp(A, Bv);
    Big diff = (c > 0) ? big_sub(A, Bv) : big_sub(Bv, A);
    long agree = 3 * (Lp - diff.len);                // a、b 已一致的十进制位数
    if (2 * agree >= p + 30) break;                  // 二次收敛：再一轮必然越过 p → 收工

    Big A2 = big_half(big_add(A, Bv));               // a' = (a+b)/2
    Big d  = (c > 0) ? big_sub(A, A2) : big_sub(A2, A);
    Big Tn = big_submul_shr(T, big_sqr(d), Lp, pn);  // T − pn·(d² >> p)：单 kernel 融合（第 14 节）
    Big K  = big_mul(A, Bv);
    Big Bn = rsqrt_ramp_cu(K, &y, agree - 9, &yn);   // b' = √(ab)：热启动的牛顿（第 9 节）

    A = keep_into(slotA, A2); Bv = keep_into(slotB, Bn);   // 结果拷入固定槽位
    T = keep_into(slotT, Tn); y  = keep_into(slotY, yn);
    pn <<= 1;
}
// 收尾：π·10^p ≈ (A+B)² · recip(4T)（最后一次牛顿倒数）

9. 牛顿的两件武器：精度爬升与热启动

牛顿开方如果只做一次，怎么写都行；但 AGM 每轮都来，加上初始化共 27 次。如果每步牛顿都在全精度上做乘法，一次开方 ≈ 23 步全尺寸大乘法，27 次开方 ≈ 600 多步——足以吞掉换算法省下的一切。两件标准武器：

9.1 精度爬升（ramping）

牛顿第 k 步只有约 2k 位是正确的——那就别在全精度上白算！把精度分成层级（每层 limb 数翻倍），最低层用 __int128 精确播种，然后逐层上爬：每层只取 K 的最高 m 个 limb 参与运算，做一步牛顿恰好把正确位数从「半层」推到「全层」。因为每层运算量是上一层的两倍，总代价 ≈ 最后一层的 2 倍——一次开方 ≈ 常数次全尺寸乘法。

9.2 热启动（warm start）

相邻两轮的 K = aₙbₙ 高位几乎一样（第 n 轮已一致 ~2n 位），而牛顿的中间量 y ≈ 1/√K 我们本来就留着——把上一轮的 y 直接当种子，跳过大部分爬升层级。最后一层照例迭代到修正量为零，天然自我纠错：就算热启动过于乐观，也只是多做一两步，不会算错。还有一个小而准的收尾优化：当修正量长度不足 y 的一半时，二次收敛保证下一步修正必然小于 1 ulp——确认步可以直接免掉。这一条把全程牛顿步数从 479 压到 443，大乘法从 1145 次压到 1062 次：

// 节选自 cuda_agm/newton.cuh 的 rsqrt_ramp_cu —— 精度爬升 + 热启动 + 免确认步
auto levels = ramp_levels(dk, ...);                 // 层级序列：limb 数逐层翻倍，直到全精度
if (warm && warm_digits >= 36) {                    // 热启动：上一轮的 y 还留着
    start = 满足 levels[i] ≤ 2×可信 limb 数 的最高层;  // 直接跳进高层
    y = 上一轮的 y 降尺度作种子;
} else {
    y = 用 __int128 在最低层精确播种;                 // 冷启动只发生一次
}
for (idx = start; idx < levels.size(); ++idx) {
    long m = levels[idx];                           // 本层工作精度：只取 K 的最高 m 个 limb
    for (int it = 0; it < max_steps; ++it) {        // 中间层 1 步；进入层/最终层迭代到收敛
        Big Ky2 = big_mul(top_slice(K, m), big_sqr(y));
        int c = -big_cmp_onehot(Ky2, 2 * hm);       // sign(B^{2h} − K·y²)：隐式 one-hot 比较
        if (c == 0) break;
        Big rr    = big_sub_onehot(Ky2, 2 * hm, c > 0);       // |B^{2h} − K·y²|
        Big delta = big_half(big_shr(big_mul(y, rr), 2 * hm)); // 牛顿修正量
        if (is_zero(delta)) break;                  // 修正为零 = 天然自我纠错的终点
        y = (c > 0) ? big_add(y, delta) : big_sub(y, delta);
        if (delta.len * 2 < y.len) break;           // 二次收敛 ⇒ 下一步修正 < 1 ulp：免掉确认步
    }
}
return big_shr(big_mul(K, y), h);                   // √K = K · (1/√K)；y 留给下一轮热启动

10. 一个线程块里的 2048 点 FFT：cuFFTDx 与 cuBLASDx 同灶合炊

从这一节起，主角登场。先把第 3 节尝过一口的「编译期菜谱」正式端上桌：Dx 库的一切配置——变换多大、什么精度、每块做几批、每线程管几个元素、目标是哪代 GPU——都用「配料 + 配料」的方式在编译期拼成一个 C++ 类型：

// 节选自 cuda_agm/fft.cuh —— 两份「编译期菜谱」
using FFT64F = decltype(cufftdx::Block() + cufftdx::Size<64>() + cufftdx::Type<fft_type::c2c>()
             + cufftdx::Direction<fft_direction::forward>() + cufftdx::Precision<double>()
             + cufftdx::ElementsPerThread<8>() + cufftdx::FFTsPerBlock<32>() + cufftdx::SM<900>());
             // ↑ 一个块做 32 批 64 点 fp64 复数 FFT；块形由此导出：(8, 32, 1)

using GEMM32 = decltype(cublasdx::Size<32, 64, 32>()        // C(32×64) = A(32×32)·B(32×64)
             + cublasdx::Precision<double>() + cublasdx::Type<type::complex>()
             + cublasdx::Function<function::MM>()
             + cublasdx::Arrangement<row_major, row_major, row_major>()
             + cublasdx::BlockDim<FFT64F::block_dim.x, FFT64F::block_dim.y, 1>()  // 与 FFT 同一块形!
             + cublasdx::SM<900>() + cublasdx::Block());
static_assert(!FFT64F::requires_workspace, "unexpected workspace");

注意 GEMM32 菜谱里的那行 BlockDim<FFT64F::block_dim…>：两位厨师要在同一个 kernel、同一个线程块里干活，就必须共用一个灶台（块形）。cuFFTDx 对 Size<64>/EPT<8>/FPB<32> 推导出块形 (8,32,1)，cuBLASDx 声明自己适配它——这一行就是「两个库合炊」的全部粘合剂。为什么要合炊？看 2048 点 FFT 的分解：2048 = 32 × 64，其中 32 点 DFT 按定义就是「32×32 常数矩阵 F₃₂ 乘数据」——把一行 2048 点重排成 32×64 矩阵后，「对 64 列各做一次 32 点 DFT」恰好是一个复数矩阵乘 C = F₃₂·X，正是 cuBLASDx 的地盘（H100 的 FP64 tensor core 自动接管）；剩下的 64 点 FFT 交给 cuFFTDx。同一份数学，两种视角：FFT 是稀疏分解的矩阵乘，矩阵乘是展开了的 DFT。

// 节选自 cuda_agm/fft.cuh 的 k_stageB（正向分支）——两个库在同一 kernel、同一块共享内存里
template<bool INV> __global__ void k_stageB(cplx* data, const cplx* fmat, long N, long rowStride) {
    extern __shared__ char smem[];
    dxcplx* row = ... + j * 2048;                                // 本块负责第 j 行
    auto [sa, sb, sc] = cublasdx::slice_shared_memory<GEMM32>(smem);   // 库切好三块共享内存
    auto a_s = cublasdx::make_tensor(sa, GEMM32::get_layout_smem_a()); // 库告诉你它想要的布局
    // ① G 级：radix-32 DFT 写成矩阵乘 —— C[k_g][s] = Σ_g F32[k_g][g] · row[s+64g]
    cublasdx::copy<GEMM32, alignment::a>(a_g, a_s);              // DFT 常数矩阵 → 共享内存
    cublasdx::copy<GEMM32, alignment::b>(b_g, b_s);              // 行数据 → 共享内存
    cublasdx::copy_wait();
    GEMM32().execute(dxcplx{1,0}, a_s, b_s, dxcplx{0,0}, c_s);   // FP64 tensor core 上的复数 GEMM
    __syncthreads();
    // ② 内 twiddle + ③ S 级：32 批 64 点 FFT（寄存器 API）
    for (unsigned i = 0; i < FFT::elements_per_thread; ++i)
        rd[i] = cmul_dx(c_s(kg, s), twiddle(-1.0, (long)s * kg, 1.0/1024));
    __syncthreads();
    FFT().execute(rd, reinterpret_cast<dxcplx*>(smem));          // 共享内存换岗给 cuFFTDx
    // ④ 外 twiddle（四步法的跨行相位）＋按存储序写回：pos = 64·k_g + k_s
    for (unsigned i = 0; i < FFT::elements_per_thread; ++i)
        row[kg * 64 + ks] = cmul_dx(rd[i], twiddle(-1.0, j * (kg + 32L * ks), 2.0/N));
}

几个值得停下来看的细节：共享内存协议——slice_shared_memory 把一块裸内存按 GEMM 的要求切成 A/B/C 三份，make_tensor + get_layout_smem_* 告诉你它想要的布局，copy<GEMM32,…> 是库提供的向量化搬运；GEMM 用完，__syncthreads() 之后同一块共享内存直接换岗给 cuFFTDx 当工作区——两位厨师前后脚用一个灶台，寸土不浪费。twiddle 精度也必须较真：所有相位分母都是 2 的幂，u/D 在二进制里精确可表示，配合 sincospi（自带精确 π 归约）做到每个 twiddle ≤1 ulp——不查表、不累积、现场算：

// e^{±2πi·u/D}，D 是 2 的幂 → u/D 在二进制里是精确值，sincospi 误差 ≤1 ulp
__device__ dxcplx twiddle(double sign2, long num, double inv_den) {
    double s, c;
    sincospi(sign2 * (double)num * inv_den, &s, &c);
    return dxcplx{c, s};
}
kernel B 有个编译开关（PSTAGE_PURE_FFTDX）能把「DFT=矩阵乘」换回单个 cuFFTDx Size<2048> 调用，两条路径全程作 A/B 对照。最终版的答案：GEMM 路径 8.76 秒、纯 FFT 路径 8.21 秒（FFT 舍入误差 0.0176 vs 0.00879——GEMM 的 32 项朴素求和约多两三层蝶形的误差）。在这条访存主导的流水线里，tensor core 的算力几乎免费、但也换不来加速——瓶颈在字节，不在乘加。GEMM 路径的价值是教学性的：演示两个 Dx 库的合炊协议与「同一变换换一种数学视角实现」的可行性；两条路径输出逐字节一致，还是第 16 节验证链的重要一环。

11. 超出一个块：四步法、三级分解，与从不重排的频谱

cuFFTDx 的天花板是「一个线程块装得下」：fp64 复数几千点。而一亿位 π 的大乘法需要 N = 228。经典解法是四步法：把长度 N = P·Q 的数组看成 P×Q 矩阵（n = Q·i + j），DFT 精确拆成两轮小 FFT 夹一个逐点相位：

X[ki+Pkj]=∑jWQjkj⋅WNjki⋅(∑ix[Qi+j]WPiki)

N 再大，就把「内层 222」当成新的积木再拆一层——三级分解 N = 222 × R（R ≤ 64），覆盖到 228：

这里有个省了大功夫的设计：kernel B 输出的频率顺序天然是「置乱」的（数位反转的混合序）。教科书做法是再加一趟全量重排恢复自然序——但我们的 FFT 只服务于卷积：正变换后马上逐点乘、马上逆变换。逐点乘不看座位号：只要正逆变换用同一套置乱序 σ，两个谱逐元素对齐即可，谁坐哪个位置根本无所谓——就像两批观众按同一张乱序座位表入座，配对关系一点不乱。于是重排一趟都不用做，逆变换就是正变换的严格镜像。唯一需要「查座位表」的场合是 2-for-1 解包（第 13 节）要找共轭伙伴 σ(N−k)，为此备一对便宜的映射函数：

// 节选自 cuda_agm/fft.cuh —— 置乱谱的「座位表」：存储序 σ ↔ 真实频率 k 的双向映射
__device__ long ki_of_s(long s, long Qin) {          // 内层存储序 s → 真实频率 k_i
    long pos = s / Qin, kq = s % Qin;
    long fpos = (pos >> 6) + 32 * (pos & 63);        // kernel B 的置乱：pos = 64·k_g + k_s
    return fpos + 2048 * kq;
}
__device__ long sigma_of_k(long k, long M, long R, long Qin) {   // 真实频率 k → 全局存储序 σ
    long kb = k / M, ka = k % M, kq = ka >> 11, r = ka & 2047;
    long pos = ((r & 31) << 6) + (r >> 5);           // ki_of_s 的逆
    return (pos * Qin + kq) * R + kb;
}
// 只在 2-for-1 解包时用到（配对 σ(k) ↔ σ(N−k)）；点乘本身从不重排数据

工程上还有一个 C++ 模板的小福利：Q 和 R 是运行期才知道的值，而 Dx 菜谱是编译期类型——一行 switch 把十来种规格各自实例化成专用 kernel，编译器对每种 Size 单独展开优化：

// 节选自 cuda_agm/fft.cuh —— 模板分发：一行 switch 实例化十种编译期 FFT
switch (Q) {                                         // Q 是运行期值，FFT 类型是编译期菜谱
    case 4:    launch_D<4>(buf, R, rowStride);    break;
    case 8:    launch_D<8>(buf, R, rowStride);    break;
    /* ... 16 / 32 / 64 / 128 / 256 / 512 / 1024 ... */
    case 2048: launch_D<2048>(buf, R, rowStride); break;
}   // 每个 case 里的 kernel 都是按 Size<Q> 单独编译优化的

到这里，完整流水线就通了：装载、三级正变换、逐点乘、三级逆变换、取整、进位——每一步一个（或几个）kernel，逻辑清晰，selftest 全绿，一亿位逐位正确。但「算得对」离「算得快」还有一整段路：这就是第 3 节那句「合同撕掉之后没人兜底」的另一面——自由是拿到了，但每一遍显存、每一次发射、每一个原子操作现在都是你自己的问题。把这条流水线交给 profiler，会看到一张意外的账单。接下来四节，就是按账单逐项施工的完整记录。

12. 优化一，先 profile，再动手：原子风暴与进位前瞻

第一直觉当然是「FFT 太慢」——毕竟它是全场最复杂的部件。但 Nsight Systems 给出的却是一张意外的账单：FFT 计算 kernel 只占约 8%；过半的运行时间花在几个不起眼的辅助 kernel 上——取整误差归约、比较、非零探测。它们的共同病灶：每个线程都对同一个全局地址做 atomicMax，1.6 亿次原子操作在同一个字上排队串行。修法是教科书式的：块内先归约，每块只留一次原子——

// 节选自 cuda_agm/fft.cuh 的 k_rint_err（修复后）—— 块内归约 + 每块一次原子
double local = 0.0;
for (; i < m; i += stride) {                        // 每线程先在寄存器里取自己的最大值
    ...
    if (e > local) local = e;
}
sh[threadIdx.x] = local;                            // 256 线程 → 共享内存
__syncthreads();
for (int s = blockDim.x / 2; s > 0; s >>= 1) {      // 块内树形归约（8 步）
    if (threadIdx.x < s && sh[threadIdx.x + s] > sh[threadIdx.x]) sh[threadIdx.x] = sh[threadIdx.x + s];
    __syncthreads();
}
if (threadIdx.x == 0) atomicMax(err, __double_as_longlong(sh[0]));   // 全 kernel 只剩几千次原子
// 修复前：1.6 亿个线程 atomicMax 打同一个地址 —— 全部串行排队

修复后，这些归约类 kernel 的单次耗时从约 10.5 毫秒回到约 0.7 毫秒的带宽下界——15 倍，它们从性能账单的榜首彻底消失。同一批改动还把第 5 节预告的进位前瞻落地成三 kernel 分块扫描（块内聚合 → 块间串链 → 应用），顺带消灭了逐趟传播里所有的主机同步。两处修完，端到端时间掉了一大截——这是全程收益最大的一批修复，而它们改的只是两类十行小 kernel。

格言一：profile 不是仪式，是破案。最显眼的嫌疑人（FFT）只占 8%，真凶是辅助 kernel 里的原子操作。没有 profiler 的优化等于闭眼开枪。

13. 优化二，少搬字节：int32 limb 与 2-for-1 打包

原子风暴平息后，账单的大头变成了纯粹的显存流量。这一批改造全是「少搬字节」：① int32 limb——limb 值域 0～999，用 int64 存纯属浪费，全部换成 int32，读写量直接砍半（卷积原始系数仍用 int64 承接，进位归一化的最后一步降回 int32）；② 装载/twiddle 融合——limb→复数的装载融进第一个转置、外层 twiddle 融进宽转置（融合思想的第一次小试牛刀，深水区在第 15 节）；③ 2-for-1 实数打包——我们的输入都是实数序列，而复数 FFT 一次天然能装两个实数组：

A(k)=Z(k)+Z¯(N−k)2, B(k)=Z(k)−Z¯(N−k)2i
// 节选自 cuda_agm/fft.cuh 的 k_pw_2for1 —— 频域解包 + 逐点乘（σ 主序重写版，见第 14 节）
for (long s1 = t; s1 < N; s1 += stride) {            // 按存储序 σ 迭代：本线程访问天然合并
    long k = ki_of_s(sa, Qin) + M * kb;              // σ → 真实频率（正向映射，便宜）
    if (k > N / 2) continue;                         // 每个共轭对由 k ≤ N/2 的一侧负责
    long s2 = sigma_of_k(N - k, M, R, Qin);          // 伙伴 bin 的座位号
    cplx z1 = Z[s1], z2 = Z[s2];
    double ax = 0.5 * (z1.x + z2.x), ay = 0.5 * (z1.y - z2.y);   // A = (Z + conj Z') / 2
    double bx = 0.5 * (z1.y + z2.y), by = 0.5 * (z2.x - z1.x);   // B = (Z − conj Z') / 2i
    Z[s1] = cplx{(ax*bx - ay*by) * invN, (ax*by + ay*bx) * invN};   // P = A·B/N
    Z[s2] = cplx{Z[s1].x, -Z[s1].y};                 // P(N−k) = conj P(k)
}

④ 牛顿免确认步（机制见第 9 节）：全程大乘法 1145 → 1062 次——最贵的操作直接少做 7%。四项合计：主要数据通路的字节流量与正变换次数都接近砍半，且不碰任何计算逻辑。

格言二：访存主导的流水线里，「省字节」就是「省时间」。int32 一刀砍掉一半读写；2-for-1 一刀砍掉一半正变换——两刀都不碰任何计算逻辑。

14. 优化三，少发射、合并访问：批量化与访存形态

14.1 kernel 发射也要钱

nsys 的下一个刺眼数字：一次一亿位计算要发射 88,169 个 kernel。每次发射有几微秒的固定开销，更糟的是发射间隙 GPU 可能在空转。几板斧下去：① 行批量流水——三级分解里「对 R 行各跑一遍内层管线」的主机循环，改成给 kernel 加 rowStride 参数＋ blockIdx.y/z 编行号，R 次发射并成 1 次；② 两阶段精确进位——逐趟有界传播换成「段内串行精确 + 段间修正」两个 kernel，任意大系数一次收干净：

// 节选自 cuda_agm/common.cuh —— 两阶段精确进位：任意大系数两个 kernel 收干净
__global__ void k_carry_seg(const int64_t* in, long n, limb_t* digits, int64_t* segout) {
    extern __shared__ int64_t sh[];                 // 64KB 动态共享内存：整块 coalesced 装载
    for (off = threadIdx.x; off < CCHUNK; off += 256) sh[off] = in[base + off];
    __syncthreads();
    int64_t q = 0;                                  // 每线程对自己的 32 个连续元素串行精确进位
    for (int k = 0; k < 32; ++k) {
        int64_t v = sh[lo + k] + q;
        q = floordiv_b(v);                          // floor 除：兼容负系数（T − pn·d²）
        sh[lo + k] = v - q * 1000;
    }
    segout[seg] = q;                                // 段出位交给下一阶段
    ...                                             // 整块写回（同样 coalesced）
}
__global__ void k_carry_fix(...) {                  // 阶段②：前段出位加进本段
    int64_t c = segout[seg - 1];
    for (k = 0; k < 32 && c != 0; ++k) { ... }      // 再传播 ≤4 个 limb 后必衰减到 ≤1
    res01[seg] = c;                                 // 残余 0/±1 → 交给前缀扫描一次收尾
}

③ one-hot 隐式操作数——牛顿步里反复出现的 Bg（1 后面 g 个 0）不再实体化成几百 MB 的数组：

// 节选自 cuda_agm/common.cuh —— one-hot 隐式操作数：
// 牛顿步天天要算 B^g − K·y²，但 B^g 是「1 后面 g 个 0」——为它实体化一个
// 几百 MB 的全零数组再读一遍，纯属浪费。改成比较/相减时「现场想象」它：
__global__ void k_sub_onehot(const limb_t* x, long nx, long pos1, long n, int sign, limb_t* out) {
    for (; i < n; i += stride) {
        int xv = (i < nx) ? x[i] : 0;
        int yv = (i == pos1) ? 1 : 0;               // B^g 的第 i 位：不用存，一个比较就有
        out[i] = sign > 0 ? (limb_t)(yv - xv) : (limb_t)(xv - yv);
    }
}   // 原始差 ∈ [−999, 999]，恰好落在借位扫描的处理域内

④ GPU 端十进制输出——最后的「大数 → 一亿个字符」此前是 CPU 串行拼接，独占 2.8 秒：

// 节选自 cuda_agm/common.cuh 的 k_to_ascii —— 十进制输出也是 GPU 的活
__global__ void k_to_ascii(const limb_t* limbs, long len, char* out) {
    for (; i < len; i += stride) {                  // 小端 limb i → 大端字符位置 (len−1−i)*3
        int v = limbs[i];
        long o = (len - 1 - i) * 3;                 // 基 1000 的好处：每 limb 恰好 3 个字符
        out[o]     = (char)('0' + v / 100);
        out[o + 1] = (char)('0' + (v / 10) % 10);
        out[o + 2] = (char)('0' + v % 10);
    }
}   // 1 亿个字符并行生成，一次 DtoH 拷回 —— output 相位从 2.85s 降到 0.29s

14.2 让每一个字节都合并

发射数降了 2/3 之后，profile 的焦点回到访存形态：同样的字节数，合并访问（相邻线程读相邻地址）与散射访问的带宽能差几倍。此时的头号热点正是 2-for-1 的解包 kernel——它按频率 k 主序迭代、按座位号 σ 散射访问，独占近三分之一的运行时间（5.48 秒）。把迭代变量换成 σ 本身（k 反查改用便宜的正向映射 ki_of_s），自身访问变成纯连续、共轭伙伴一侧也呈镜像连续段——一个 kernel 省下 3 秒多。配上扫描装载的共享内存整块化、T − pₙ·(d²≫p) 三合一 kernel 等零碎收尾。

相位视角（本节两批改造前后）：carry 1.84 → 0.98 秒，fft 14.7 → 9.52 秒，output 2.85 → 0.28 秒。

格言三：发射数和访存形态是「整片数组」心法的另一半。把循环改成整片运算只是开始；整片运算发射多少次、每个线程摸到的地址挨不挨着，决定它跑多快。

15. 优化四，深度融合：设备端库的独门优势

至此，每次大乘法的中段仍是一条十几个 kernel 的链：纯搬运 kernel（转置/置换/装载/取整）与计算 kernel（FFT/GEMM/逐点乘）交替出现，每个都是一遍全量显存读写——228 点时一遍就是 4GB。现在兑现第 3 节的伏笔：既然 FFT 是你 kernel 里的一行，它前后的装载/写回相位就是你的代码——把相邻的搬运 kernel 折进计算 kernel 的装载/写回相位，发射和整遍带宽一起省。这样的动词在主机调用模型里根本不存在（那里连「装载时顺手转个类型」都要走回调的窄门），而在设备端模型里只是改自己 kernel 里的一个循环。判断能不能融只有一条法则：融合后一个线程块的工作集必须装进共享内存（H100 上限 227KB）。逐一排查：

融合点块工作集判定
F2：外层 twiddle 转置 + R 点 FFT（正逆两向）R×32 ≤ 64×32 复数 ≈ 34KB可行
F3：逆变换尾部两连置换 + 取整 + 误差归约32×32 tile ≈ 16KB可行
F1：入口装载 + 首个置换32×32 tile ≈ 16KB可行
kernel B 两侧的行内转置32 行 × 2048 点 ≈ 1MB不可行（> 227KB）

F2 是最大的一块。融合前：twiddle 转置 kernel 写出 4GB，R 点 FFT kernel 再读回 4GB；融合后一个 kernel 里走完——twiddle 在「反正也要读」的装载相位上白送，块内转置经共享内存 slab 完成（[R][33] 布局，多出的一列专治 bank 冲突），FFT 在寄存器上就地执行，结果直接按 σ 序落显存：

// 节选自 cuda_agm/fft.cuh 的 k_stageR_fused（融合改造 F2，正向分支）
// block = 32 个连续 s 列 × 全部 R 行；twiddle 转置 + R 点 FFT 一个 kernel 完成
template<unsigned Rv, bool INV>
__global__ void k_stageR_fused(const cplx* in, cplx* out, long M, long N, long Qin) {
    extern __shared__ char smem[];
    dxcplx* slab = reinterpret_cast<dxcplx*>(smem);          // [Rv][33]，+1 列防 bank 冲突
    for (int idx = tid; idx < Rv * 32; idx += nthreads) {    // ① 装载：按显存行 coalesced 读
        int b = idx >> 5, sc = idx & 31;
        cplx v = in[(long)b * M + s0 + sc];
        if (b > 0) v = v * W_N(-b * ki_of_s(s0 + sc, Qin));  // twiddle 白送：反正这一遍要读
        slab[idx + (idx >> 5)] = v;                          // slab[b][sc]
    }
    __syncthreads();
    for (i...) rd[i] = slab[e * 33 + sy];                    // ② 列方向取进寄存器（转置）
    __syncthreads();                                         //    slab 用毕，FFT 可复用同一块 smem
    FFT().execute(rd, reinterpret_cast<dxcplx*>(smem));      // ③ R 点 FFT（cuFFTDx）
    for (i...) out[(s0 + sy) * Rv + e] = rd[i];              // ④ 按 σ 序写回（R 连续 run）
}   // 融合前：k_transpose2_twr 写 4GB + k_stageR 再读 4GB —— 全省了

F3 处理逆变换尾部：那里原本是两个背靠背的纯转置 kernel 加一个取整 kernel（三遍全量读写）。两个置换的复合仍是一个置换——推一下下标代数：in[b·M + j·2048 + i] 应落到 raw[T·i + t]（T = R·Q，t = R·j + b），恰好读侧写侧都能做到合并访问；取整与 0.25 红线的误差归约顺势成为写出相位：

// 节选自 cuda_agm/fft.cuh 的 k_perm_rint（融合改造 F3）
// 逆变换尾部两个背靠背转置的复合恰是一个置换：in[b·M + j·2048 + i] → raw[T·i + t]
// （T = R·Q，t = R·j + b）。取整 + 误差归约作为写出相位，int64 直接落地。
__global__ void k_perm_rint(const cplx* in, long M, long R, long T, long m,
                            int64_t* raw, unsigned long long* err) {
    __shared__ cplx tile[32][33];
    for (dt...) {                                            // 读侧：每 t 行 32 个连续 i（合并）
        long b = t % R, j = t / R;
        tile[dt][threadIdx.x] = in[b * M + j * 2048 + i0 + threadIdx.x];
    }
    __syncthreads();
    for (dy...) {                                            // 写侧：每 i 行 32 个连续 n（合并）
        long n = T * (i0 + dy) + t0 + threadIdx.x;
        cplx v = tile[threadIdx.x][dy];
        raw[n] = llrint(v.x);                                // 取整直接写 int64 原始系数
        local = max(local, |v.x − rint|, |v.y|);             // 0.25 红线的素材
    }
    块内归约 local → atomicMax(err);                          // 沿用第 12 节的单原子模式
}   // 融合前是三个 kernel：transpose_b + transpose2 + k_rint_err

F1 是 F3 的镜像（入口装载 + 首个置换复合），不再展开。三批改造各自独立落地、各自过完整验证阶梯（第 16 节）——恒等式测试在最小的三级规模 223 上秒级暴露任何置换/twiddle 笔误，这是「先小后大」的验证纪律第无数次领工资。三级流水线的 kernel 数：正变换 7 → 5，逆变换 8 → 5。成绩：

深度融合（1 亿位）融合前融合后变化
GEMM 路径 墙钟11.68 秒8.76 秒−25%
纯 FFT 路径 墙钟11.15 秒8.21 秒−26%
fft 相位9.52 秒6.92 秒−27%
kernel 发射数28,54825,542−11%
全文第二重要的一句话：库的 kernel 边界是墙时，你只能优化墙与墙之间；库的 kernel 边界是你画的线时，整条流水线都是优化对象。这套流水线的性能工程里，原子归约、省字节这类改造在任何执行模型下都能做，而这最后一段深度融合——以及第 13 节的装载/twiddle 融合——只属于设备端路线。同时诚实记录边界的另一侧：kernel B 两侧的转置融不动（1MB ≫ 227KB），剩余时间里它们仍占一份——共享内存的大小，就是这条独门优势的射程。

收束一下这条设备端工作流积累的能力清单：① 融合控制权——本节整节，任何主机端库给不了；② 零主机开销——没有 plan、没有句柄、没有库状态机，第 9 节那条每轮 0.31 秒的水平线就是证词；③ 单 kernel 组合能力——kernel B 里 GEMM 与 FFT 共用一块共享内存接力（第 10 节）；④ 编译期定制——每种规格单独优化的专用 kernel，没有运行期分支税。还剩什么没做？实数半谱变换（复数变换只算半张谱）估计还能再省 2–3 秒，但需要重做整个置乱序设计，记在「未来工作」里。

16. 正确性验证

手搭 FFT 最怕「悄悄算错」。三重证据链：

① 与公开基准逐位比对：输出与 MIT 镜像的 π 十亿位文件（pi-billion.txt）的前 100,000,000 位全部一致（基准文件本身先核对了前 50 位与第 762–767 位的费曼点六连 9）。

② 两条数学路径互为对拍：kernel B 的 GEMM 分解路径与单个 2048 点 FFT 路径（第 10 节的编译开关）走完全不同的计算图——置乱序不同、twiddle 组织不同、舍入误差谱不同——两份一亿位输出却逐字节一致。同一套置换/进位/牛顿代码若有 bug，很难在两条路径下产出同一串错误数字。

③ 全程红线 + 分级自检梯子：每次乘法查 0.25 红线（第 6 节）；上线前六级阶梯，每级只暴露一类新风险——加减进位 ↔ __int128 对拍；schoolbook ↔ FFT 交叉；恒等式覆盖全部 FFT 规格；牛顿窗口检查（含热启动重入）；小规模 π 逐位比对；最后才是一亿位。其中恒等式一级是所有工程改造的安全网——第 12–15 节的每一批改动都要先过它：

// 节选自 cuda_agm/pi_agm.cu 的 selftest —— S2：恒等式覆盖每一种 FFT 规格
// (X+Y)² = X² + 2XY + Y² 同时检验：平方路径、2-for-1 路径、进位、全部 N/Q/R 组合
for (long L : {4000, 8000, ..., 2080000,        // 两级：N = 2^13 … 2^22（每一种 Q）
               4100000, 41000000, 82000000}) {  // 三级：N = 2^23 / 2^27 / 2^28（R = 2/32/64）
    Big X = rand_big(L), Y = rand_big(L);
    Big lhs = mul_fft_path(big_add(X, Y), ..., /*square=*/true);
    Big rhs = big_add(big_add(sqr(X), sqr(Y)), 2 * mul(X, Y));
    if (big_cmp(lhs, rhs) != 0) FAIL;           // 任何一处置换/twiddle/进位写错都藏不住
}

17. 结语

几件值得带走的事：

• 算法先于实现：二次收敛把一亿位压成 26 轮大运算——选对算法，问题的形状就先对了一半。
• 快算法把压力转移给基础设施：牛顿的特殊数值形态逼出进位前瞻，一亿位的规模逼出 228 点三级 FFT。
• profile 不是仪式：「真凶不是嫌疑人」上演了三次（原子风暴、发射计数、σ 散射），全靠 nsys 定罪。
• 省最多的字节，而不是最多的指令：int32、2-for-1、融合，三次大幅加速全是带宽账。
• 库的 kernel 边界是你画的线：把数学库请进 kernel，买到的不是更快的 FFT，而是对装载与写回相位的支配权——本文性能工程里最深的那一层，全部长在这份支配权上。

Gauss–Legendre/Brent–Salamin、FFT 大数乘法、牛顿倒数/开方、进位前瞻均为高精度计算与硬件设计的经典方法；kernel B 的 GEMM+FFT 分解参考 MathDx 官方示例。
