1. Introduction: What This Tutorial Teaches

There are two very different postures for using a math library on the GPU. The first is to place a phone call: you call a host-side library such as cuFFT — one function goes in, and the library finishes the job for you, launching kernels and handling scheduling on its own while you wait for the result. The second is to invite the library into your own kernel: cuFFTDx (the device-side FFT library in NVIDIA MathDx) finishes nothing on your behalf; it simply turns "a small FFT inside one thread block" into one ordinary statement in your kernel code — the data lives in your own registers and shared memory, and every byte of movement into and out of that statement is yours to manage.

This tutorial walks the second posture from start to finish: how to assemble the transform's "recipe" in Python, how to execute it inside a CUDA kernel, how to stitch small "fits-in-one-block" transforms into a big 228-point FFT, and — most important of all — how much performance "managing the data traffic yourself" is really worth. To give the answer some weight, we chose a running example that is punishing enough: computing π to 100,000,000 digits on a single H100, with all big-number arithmetic implemented from scratch. 2. Parallelism 101: From "Loops" to "Whole Arrays"

CPU = a few PhD students
Few cores, but strong ones: good at complex logic where "each step depends on the previous one"; running a for loop means one cell at a time.
GPU = ten thousand schoolchildren
This H100 has over ten thousand CUDA cores; each knows only simple arithmetic, but they can all start working at once.

The smallest unit of organization in a GPU program is the thread; threads are grouped into thread blocks (most blocks in this article hold 256 threads). Threads in the same block can cooperate through a small patch of ultra-fast shared memory, while blocks can only meet one another through global memory. The most important sentence in this article: every big-number operation that follows will be written as "an operation over the whole limb array" — that is what makes it truly run on the GPU. Any operation that cannot be written this way (such as the inherently serial digit-by-digit carry) must be recast into a parallel equivalent — the subject of Section 5 and Section 13.

This lesson has a second half, and it is this article's distinctive angle: once loops become whole-array operations, every "whole-array" pass is one full round of global-memory reads and writes. For 100-million-digit π, the intermediate arrays easily run to 4GB, and every extra trip through global memory costs a few extra milliseconds. Whoever controls the number of passes controls the performance — and that is precisely the watershed between the two library-usage models in the next section.

3. Phone Call, or Into the Kitchen

Start with the familiar host-call model. To run one "transform — pointwise multiply — inverse transform" pipeline with cuFFT, the host code typically looks like this:

# Sketch: one big-number multiplication under the host-call model
buf = to_complex(a_limbs)                      # ① Preprocess: launch a kernel of your own first — one full-length read/write
fa = cp.fft.fft(buf)                           # ② Phone call: the library launches its own kernel for the forward transform
fb = cp.fft.fft(to_complex(b_limbs))           # ③ The second operand: another phone call
spec = fa * fb                                 # ④ Pointwise multiply: cupy launches yet another kernel for this one line
raw = cp.fft.ifft(spec)                        # ⑤ Inverse transform (one more phone call)
digits = cp.rint(raw.real)                     # ⑥ Rounding: yet another full-length read/write
carry_kernel[blocks, 256](digits, out_limbs)   # ⑦ Per-digit carries: your own Numba kernel finishes the job
# Between every step the data crosses global memory in full; not one line can be squeezed inside the library's kernels

The contract of this model is clear: you build the plan on the CPU (the library's execution-plan handle), put the data in global memory, place a phone call, and the library launches its own kernel to finish that step. Painless, mature, fast — but two restrictions are written into the contract in permanent ink: ① between every step, the data must lie back in global memory in full (library kernels accept nothing but global-memory pointers for input and output); ② you cannot insert a single line of code inside a library kernel (want to do a type conversion in passing while loading? Sorry — first launch a kernel of your own to convert the types).

The device-inline model tears up both clauses of that contract. cuFFTDx enters Python through nvmath-python's device-side API: at construction time you declare the transform you want by "assembling the ingredients as keyword arguments", and it hands you a descriptor; inside your own kernel, that descriptor's execute() completes an FFT within one thread block on the spot — no new kernel is launched, and the data stays in your registers and shared memory the whole way:

4. Enter the Running Example: 100 Million Digits of π, and Why It Stresses Everything

5. Math Building Blocks I: limb Big Numbers, Carries, and the 999…9 Reef

Representation. Chop the big integer into many small numbers in the range 0–999; each one is called a limb, and the whole thing is an int32 array (least significant limb first). We choose base B=1000 (=10³) because the final answer must come out in decimal: each limb is exactly 3 decimal digits, so output costs nothing. A 100-million-digit π is an array of about 33.33 million limbs (a bit over 130MB).

Addition and carries. The grade-school method — "add position by position, carry when you hit a thousand" — is inherently serial (the carry at position k must wait for position k−1), which puts it squarely in the crosshairs of Section 2's whole-array mindset. On the GPU we split it in two: first add the whole array at once (some limbs temporarily exceed 999 — never mind), then propagate the carries in parallel. That propagation step hides the most dangerous reef in the entire project: Newton's iteration (Section 7) systematically manufactures numbers like a 1 followed by hundreds of thousands of 0s (or 999…9 when it lands just short) — a single +1 carry then has to travel the full length of that chain of 0s/9s. The naive parallel scheme — "many passes, each pushing the carries one position" — degenerates to O(n) passes on such inputs: in one multiplication at the million-digit scale, carry propagation alone takes 166,679 passes — well over a hundred thousand kernel launches, with launch overhead by itself running to tens of seconds. Completely unusable.

The fix has the same pedigree as the classic answer in hardware adders — carry-lookahead. The observation: a residual carry can only "pass through" a 999 (a borrow, only through a 000); any other digit absorbs it. So "will position i receive a carry" = "does the nearest non-999 position below i fire" — and that is a prefix scan: it computes every position at once, and no matter how long the chain, one pass goes all the way:

# Serial carry (one step at a time)       # Parallel prefix (one scan, all the way)
carry = 0                                 # when the residual carry ∈ {0,±1}:
for i in range(n):                        #   999 lets a carry pass through (propagate)
    v = s[i] + carry                      #   ≥1000 always produces a carry (generate)
    s[i] = v % 1000                       # (g,p) form an associative semigroup → blocked prefix scan
    carry = v // 1000                     # three CUDA kernels: in-block reduce → cross-block chain → apply
Lesson: the "average-case" performance of a vectorized primitive can hide an O(n) worst case — and numerical algorithms are exactly the workloads that systematically manufacture worst-case inputs. Fast algorithms shift the pressure onto the infrastructure.

Subtraction uses the symmetric borrow formulation; comparison scans from the most significant end for the first differing limb; "divide by 2" has a slick parallel trick: base 1000 is even, so the result at position i = d[i]/2 + 500·(d[i+1] is odd) — every limb computed in one shot, no propagation needed. The CUDA implementations of all these primitives live in common.py. The final form of carrying (the exact two-phase carry) arrives in Section 13.

6. Math Building Blocks II: FFT Multiplication and the 0.25 Red Line

Think of a big number as a "base-1000 polynomial," and multiplying two numbers = a convolution of their coefficients. Naive convolution is O(n²) — at 33.33 million limbs that is roughly 1015 multiply-adds, tens of minutes of computing, and we have to do it 1062 times. The convolution theorem offers a shortcut:

x⋅y=F−1(F(x)⊙F(y)), ☉ denotes pointwise multiplication

How does floating point produce exact integers? The FFT runs on float64 internally, and float64 represents integers exactly up to 253 ≈ 9×1015. At 100 million digits the largest convolution coefficient is about n·999² ≈ 3.34×107×106 ≈ 3.3×1013 — still a factor of about 270 below the limit. But that only means "the coefficients themselves fit"; the FFT's tens of millions of floating-point multiply-adds also accumulate rounding error, so every multiplication is guarded by a safety valve: check how far each floating-point result sits from the nearest integer, and the moment that distance exceeds 0.25, raise an error and abort — never silently compute a wrong answer:

# Excerpt from pi-numba/fft.py — the 0.25 red line on every multiplication: residuals accumulate device-side, checkpoints read back and pass judgment
def fft_error_read():
    bits = int(g_err.get()[0])
    cp.cuda.runtime.memsetAsync(g_err.data.ptr, 0, 8, 0)
    return np.frombuffer(np.int64(bits).tobytes(), dtype=np.float64)[0]

def fft_error_checkpoint(where=""):
    # Residuals accumulate device-side; checkpoints read back and pass judgment (the red line is never lifted)
    err = fft_error_read()
    if err > g_cnt.fft_err_max:
        g_cnt.fft_err_max = err
    assert err < 0.25, f"FFT rounding error too large: {err} at {where}"

The measured maximum error across the full 100-million-digit run is 0.00781 — an order of magnitude below the red line, yet already revealing: base 1000 is nearing its limit at the 228-point scale (ten times bigger and we would have to switch to base 100). That is precisely the red line's value: it turns "precision is about to run out" into an explicitly observable quantity, rather than a string of wrong digits quietly showing up one day.

Very small multiplications (where the product of the two operands' lengths — i.e., the number of O(n²) multiply-adds — is ≤ 1.6×107) go straight to a schoolbook kernel, dodging the FFT's fixed overhead — the low levels of Newton's precision ramping are all small multiplications of this kind.

7. Math Building Blocks III: Newton's Iteration — Division and Square Roots from Multiplication Alone

Computing π also calls for division and square roots, but we only want to use the multiplication we have already built. Newton's iteration makes that possible. To compute the reciprocal 1/T:

xn+1=xn⁢(2−T⋅xn)

To compute the reciprocal square root 1/√K (taking a square root is then just √K = K·(1/√K)):

yn+1=yn⁢(3−K⋅yn2)2

Both iterations involve nothing but multiplication, subtraction, and "division by 2" (the parallel halving from Section 5), and both exhibit quadratic convergence — each iteration step doubles the number of correct digits: 15 → 30 → 60 → … → 108, so starting from a float64 seed, about 23 steps get you all the way there. In the integer world, 1/√K is represented in the fixed-point scaled-up form y ≈ Bh/√K (h is slightly larger than K's limb count; the implementation uses h = len(K)+2, so K·y² ≈ B2h, which is exactly where the residual B2h − K·y² in Section 9's code comes from), and "dividing by a power of B" is simply dropping an entire low-order block of the array — exact and zero-cost.

Remember "digits double every step" — Section 8's choice of algorithm and Section 9's two weapons are all built on this one property.

8. The Star Algorithm: Gauss–Legendre and Fixed-Point Scaling

With the infrastructure in place, which formula do we use to compute π? The most famous candidate, the Leibniz series, is a cautionary tale:

π4=1−13+15−17+⋯

A million terms buy you just 6–7 digits. History offers series that converge far faster (the good ones contribute a dozen-plus digits per term), but all series share one ceiling: the number of correct digits is proportional to the number of terms (linear convergence), and merging an ocean of tiny terms is exactly the fragmented workload GPUs like least. Gauss–Legendre (Brent–Salamin, 1976) does not sum a series at all — it repeatedly applies a set of extremely simple transformations:

a0=1,b0=12,t0=14,p0=1
an+1=an+bn2, bn+1=anbn, tn+1=tn−pn(an−an+1)2, pn+1=2pn
π≈(an+bn)24tn

a takes the arithmetic mean and b the geometric mean, and the two close in on each other at an astonishing rate (their common limit is called the "arithmetic–geometric mean", AGM). The key property is the same one Newton offers: quadratic convergence, with matching digits doubling every round — 100 million digits take only 26 rounds, and each round is a big whole-array operation, exactly the workload shape GPUs love:

The price is equally explicit: every round takes one square root at full 100-million-digit precision (b′=√(ab)), and the wrap-up adds one division at the same precision. Whether this algorithm is fast comes down entirely to how fast big-number multiplication is and how cleverly Newton's iteration is run — which channels all the pressure straight onto our infrastructure.

Fixed-point scaling solves the "real numbers into an integer library" problem: every quantity is multiplied by 10p and stored as an integer (p = 100,000,050, a multiple of 3 → exactly 33,333,350 limbs). Every real-number operation then corresponds strictly to one integer operation:

9. Newton's Two Weapons: Precision Ramping and Warm Starts

If Newton's square root ran only once, any implementation would do; but AGM invokes it every round — 27 times in all, counting initialization. If every Newton step multiplied at full precision, one square root would cost ≈ 23 full-size big multiplications, and 27 square roots ≈ 600-odd — enough to swallow everything the better algorithm had saved. Two standard weapons:

9.1 Precision ramping

At Newton step k, only about 2k digits are correct — so don't waste work at full precision! Split the precision into levels (the limb count doubling per level), seed the lowest level exactly with Python big integers (math.isqrt), then climb level by level: each level takes only the top m limbs of K into the computation, and one Newton step pushes the correct digit count from "half a level" to "a full level" exactly. Because each level costs twice as much as the one before, the total cost is ≈ 2× the last level alone — one square root ≈ a constant number of full-size multiplications.

9.2 Warm starts

Across two adjacent rounds, the high digits of K = aₙbₙ are nearly identical (by round n they already agree to ~2n digits), and we keep Newton's intermediate quantity y ≈ 1/√K around anyway — so feed the previous round's y in directly as the seed and skip most of the ramp levels. The last level still iterates until the correction reaches zero, which makes it naturally self-correcting: even if the warm start was too optimistic, it merely costs an extra step or two and can never produce a wrong answer. There is also one small but exact finishing touch: once the correction is shorter than half the length of y, quadratic convergence guarantees the next correction is below 1 ulp — the confirmation step can be skipped outright. This one trick cuts the total Newton steps from 479 to 443 and the big multiplications from 1145 to 1062:

10. A 2048-Point FFT Inside One Thread Block

This is where the star of the show takes the stage. Let's start by formally serving the "recipe" we sampled back in Section 3: everything cuFFTDx needs configured — how big the transform, what precision, forward or inverse, how many batches per block, execution at block level — is, in nvmath-python, just a table of keyword arguments, and one ordinary Python function call assembles them into an FFT descriptor. Every FFT recipe in the whole project is cooked up by the same little factory:

# Excerpt from pi-numba/fft.py — the "recipe" factory: one function call assembles the ingredients, returns an FFT descriptor
from nvmath.device import FFT as DxFFT

def _dx(size, inv, fpb, ept=None):
    kw = dict(fft_type="c2c", size=size, precision=np.float64,   # ingredients: complex transform, size, fp64
              direction=("inverse" if inv else "forward"),        # forward / inverse
              ffts_per_block=fpb, execution="Block")              # batches per block, block-level execution
    if ept is not None:
        kw["elements_per_thread"] = ept                           # leave unset and the library picks the optimum
    return DxFFT(**kw)                                            # ← one recipe; launches no kernel

The biggest difference from the old "build a plan, then place a phone call" model: this descriptor does nothing on your behalf — it is a recipe. Construction happens at Python run time, but the moment it lands in your hands, the recipe tells you everything about how the kitchen should be set up: storage_size is how many register slots each thread must reserve, elements_per_thread / stride are the pass count and interleave stride of the load loop, value_type is the element type of the register and shared-memory arrays, and block_dim / shared_memory_size are the block shape and workspace size at launch. These traits are captured by closure into the CUDA kernel as Python constants — the moment Numba compiles, they all fold into constants, and the generated machine code is no different from hard-coding the numbers.         rd = cuda.local.array(shape=(st,), dtype=vt)   # the data lives in your registers
        sm = cuda.shared.array(shape=(0,), dtype=vt)   # dynamic shared memory: the library's workspace
        j = cuda.blockIdx.x                            # this block handles row j
        base = cuda.blockIdx.y * rowStride + j * FFT_P
        invN2 = 2.0 / N
        for i in range(ept):                           # load phase: a loop you wrote
            e = cuda.threadIdx.x + i * stride
            if e < FFT_P:
                v = data[base + e]
                if IS_INV:
                    sn, cs = libdevice.sincospi((j * e) * invN2)
                    v = v * complex(cs, sn)            # inverse: multiply the twiddle in passing on load
                rd[i] = v
        fft.execute(rd, sm)                            # ← the library, right here: a 2048-pt FFT in one line
        for i in range(ept):                           # write-back phase: also a loop you wrote
            e = cuda.threadIdx.x + i * stride
            if e < FFT_P:
                v = rd[i]
                if not IS_INV:
                    sn, cs = libdevice.sincospi(-(j * e) * invN2)
                    v = v * complex(cs, sn)            # forward: multiply the twiddle in passing on write-back
                data[base + e] = v

    return k_stageB, fft                               # hand the kernel and the recipe to the launcher together

A few details worth pausing over: the load shape is a registration form set by the library — each thread owns ept elements, interleaved with its peers at stride (e = threadIdx.x + i·stride); load and write-back must copy this form verbatim or execute won't honor it (if e < FFT_P is a general guardrail against ept·stride running past the end of the row under some configurations); fft.execute(rd, sm) is one line — a block-level device-side call compiled into your kernel, launching no new kernel: the 2048-point butterflies finish in place in shared memory and the result returns to the registers rd; the direction fork vanishes inside the factory — IS_INV is a closure constant: the forward and inverse kernels compile separately, and the machine code holds not a single run-time branch:

Twiddle precision demands the same rigor: every phase here has a power-of-2 denominator — invN2 = 2.0/N is exact in binary, j·e is an integer no larger than 222, and their product loses not a single bit; libdevice.sincospi comes with exact π reduction and delivers sine and cosine in one call (the Python side unpacks them straight into sn, cs, then finishes the rotation with v * complex(cs, sn)), so each twiddle carries an error of ≤1 ulp — no lookup tables, no accumulation, computed on the spot. This "cheap rigor" is the least worrisome line item in Section 6's 0.25 red-line budget.

The final step: the recipe turns around and tells you how big a kitchen to open — you compute none of the launch parameters yourself; take them exactly as written:

11. Beyond One Block: the Four-Step Method, Three-Level Decomposition, and a Spectrum That Is Never Reordered

cuFFTDx's ceiling is "whatever fits in one thread block": a few thousand fp64 complex points. But the big multiplication behind 100-million-digit π needs N = 228. The classic answer is the four-step method: view an array of length N = P·Q as a P×Q matrix (n = Q·i + j), and the DFT splits exactly into two rounds of small FFTs sandwiching a pointwise phase:

X[ki+Pkj]=∑jWQjkj⋅WNjki⋅(∑ix[Qi+j]WPiki)

When N grows past that, treat the "inner 222" as a new building block and split one more level — three-level decomposition N = 222 × R (R ≤ 64), covering up to 228:

Here lies a design choice that saved enormous effort: the frequency order kernel B emits is naturally "scrambled" (a digit-reversed mixed order). The textbook fix is an extra full-array reordering pass to restore natural order — but our FFT serves only convolution: right after the forward transform comes the pointwise multiply, then immediately the inverse transform. The pointwise multiply never looks at seat numbers: as long as the forward and inverse transforms use the same scrambled order σ, the two spectra line up element by element, and who sits in which seat simply does not matter — like two audiences seated by the same shuffled seating chart: the pairing stays perfectly intact. So not a single reordering pass is needed, and the inverse transform is a strict mirror of the forward one. The only place that has to "consult the seating chart" is 2-for-1 unpacking (Section 12), which must find the conjugate partner σ(N−k); for that we keep a pair of cheap mapping functions:

# Excerpt from pi-numba/fft.py — the scrambled spectrum's "seating chart": bidirectional map, storage order σ ↔ true frequency k
@cuda.jit(device=True)
def ki_of_s(s, Qin):
    pos = s // Qin
    kq = s - pos * Qin
    return pos + 2048 * kq

@cuda.jit(device=True)
def sigma_of_k(k, M, R, Qin):
    kb = k // M
    ka = k - kb * M
    kq = ka >> 11
    r = ka & 2047
    return (r * Qin + kq) * R + kb

Engineering throws in one more small perk, a "factory + cache": Q and R are values known only at run time, while the recipe is a construction-time constant — so look up the cache once per spec, and each of the dozen or so Sizes specializes into its own dedicated kernel, with Numba compiling and optimizing every one of them separately:

13.1 Kernel launches cost money too

13.2 Make every byte coalesced

With the launch bill now on the table, the profile's focus returns to memory-access shape: for the same byte count, coalesced access (adjacent threads reading adjacent addresses) and scattered access can differ several-fold in bandwidth. The number-one hotspot at this point was precisely the 2-for-1 unpacking kernel — iterating in frequency-k-major order and scattering its accesses by seat number σ, it at one point monopolized nearly a third of the runtime. Switching the iteration variable to σ itself (replacing the reverse lookup of k with the cheap forward map ki_of_s) makes its own accesses purely contiguous, and the conjugate-partner side falls into mirrored contiguous segments as well — the top hotspot stepped down on the spot. Round this off with shared-memory blocking of the scan loads, the T − pₙ·(d²≫p) three-in-one kernel, and other odds and ends.

Maxim 2: Launch counts and memory-access shape are the other half of the whole-array mindset. Turning loops into whole-array operations is only the beginning; how many launches those operations take, and whether the addresses each thread touches are adjacent, decide how fast they run.

14. Python's Launch Tax

14.1 The dispatcher: a 40 µs customs office on every launch

The previous section counted the number of launches; this section counts their unit price. Numba's kernel object is a dispatcher: every call configures the grid parameters again, redoes type matching down in the C layer, and re-parses __cuda_array_interface__ for each cupy array argument — about 40 µs per launch in total. This is a customs office unique to the interpreter's world, levied on top of the GPU-side launch latency. A full run at 1e8 digits fires 23,251 kernel launches, so the tax bill totals on the order of 1 second: the kernels didn't change and the launch count barely moved (Figure 11) — all the money is spent at the Python↔driver border.

# Excerpt from pi-numba/common.py — thin launch: pre-specialize + direct-wire overloads,
# skipping the dispatcher's per-call configure/typeof
def prime(kern, *args):
    """Lazy specialize + cache; returns a compiled _Kernel ready for direct .launch."""
    krn = _spec_cache.get(kern)
    if krn is None:
        spec = kern.specialize(*args)
        krn = next(iter(spec.overloads.values()))
        _spec_cache[kern] = krn
    return krn

def launch(kern, grid, block, smem, *args):
    g_cnt.launches += 1
    args = tuple(_nb_twins.get(id(a), a) for a in args)   # resident buffer → numba twin
    prime(kern, *args).launch(args, _d3(grid), _d3(block), core_stream0(), smem)

The countermeasure is not to bypass Numba but to make its slow path run only once: on the first call, prime() specializes and caches the compiled _Kernel; from then on, launch() direct-wires into its .launch entry point, and all three formalities — configure, type matching, argument parsing — are out of the picture. The precondition is written in the source comments: every kernel's call signature stays stable for the whole run (only the dtype/ndim of arena views enter the type, and the scalar side never forks between int and float), with full selftest coverage as the safety net.

14.2 Resident twins and pinned readback

With the direct wire in place, the costliest step left on the launch path is parsing the CAI for every cupy argument — yet for resident buffers (the FFT workspaces, g_pos, objects that never change for the whole run), that parse yields the same result every time. register_twin() wraps them once as numba device arrays; launch() swaps in the twin by id, takes the fast is_device_memory branch, and CAI parsing drops to zero. The same medicine also cures host readback: read_pos() does an async copy through a preallocated pinned buffer, sparing the temporary allocation on every readback.

15. Deep Fusion: the Device-Side Library's Exclusive Advantage

Up to this point, the middle stretch of every large multiplication is still a chain of a dozen-plus kernels: pure data-movement kernels (transpose / permutation / load / rounding) alternating with compute kernels (FFT / pointwise multiply), each one a full pass of GPU-memory reads and writes — and at 228 points a single pass is 4GB. Now the foreshadowing from Section 3 pays off: since the FFT is a line inside your kernel, the load and write-back phases around it are your code — fold the adjacent data-movement kernels into the compute kernels' load and write-back phases, saving launches and whole passes of bandwidth in one stroke. That verb simply does not exist in the host-call model (there, even "convert a type in passing while loading" has to squeeze through the narrow door of a callback), while in the device-side model it is nothing more than editing a loop in your own kernel. There is exactly one rule for judging whether a fusion is possible: after fusion, one thread block's working set must fit in shared memory. Going down the list:

The trick that computes only the high window is a reversal: reverse the high prefix of each operand, and the high window of the original product becomes exactly the low window of the reversed product — and the low window depends only on the convolution of the two short prefixes, so the FFT size drops from Nfull to Nrev. At the 100-million-digit workhorse tier that one drop is worth three levels (228 → 225, an FFT roughly 8× cheaper); if the size will not drop (Nrev ≥ Nfull), we fall back to the full multiplication outright — better not to bother than to bother for nothing.

The remaining question is exactness — the truncated low limbs can still invade the high window through a long carry chain; Section 5's 999…9 reef is back in a new disguise. The treatment here is this article's best demonstration of the relationship between "fast" and "accurate": instead of leaving a few extra guard limbs out of habit, turn the uncertainty into an interval, then verify that the interval collapses. The unknown incoming carry at the truncation point has a strict upper bound U (derived from the convolution-coefficient upper bound min(nx,ny)·999²; guard is 2 + logBU, in practice 5–6 limbs); both ends of the interval [0, U] are pushed through the guard window together, and the truncated result is accepted only if the two ends converge to the same carry value at the window's exit:

# Excerpt from pi-numba/fft.py — big_mul_high_proto: the strict truncated-carry interval [0,U] crosses the guard window at both ends; no collapse means fallback
    cmax = min(nx, ny) * 999 * 999
    U = (cmax + (LIMB_B - 2)) // (LIMB_B - 1)
    guard = 2
    t = U
    while t:
        guard += 1
        t //= LIMB_B
    ...
    hg = draw[wanted:wanted + guard].get()          # guard window (host-side interval crossing)
    qlo, qhi = 0, U
    for i in range(guard):
        c = int(hg[guard - 1 - i])
        qlo = (c + qlo) // LIMB_B
        qhi = (c + qhi) // LIMB_B
    if qlo != qhi:
        _mlog(f"{semantic},nx={nx},ny={ny},shift={shift},guard={guard},used=0,reason=carry_interval,lo={qlo},hi={qhi}")
        g_cnt.add("middle_fallback")
        return strip(big_shr(big_mul(x, y), shift))

So this is an exact middle product, not an approximate high product — every limb matches the full multiplication digit for digit, and the 0.25 red line stays on duty on the shrunken FFT. Over the full 100-million-digit run: 354 windows took the shrunken FFT, while 116 calls fell back safely because the shape was wrong, the size too small, or the interval not unique; the adversarial inputs of all 999s and ultra-long carry chains (Section 19's middlebench) owe their limb-for-limb correctness precisely to that fallback. The single largest gain on the whole ladder comes from an FFT core in which not one line was changed.

Maxim 4: The fastest multiplication is the one you never compute. Section 12 saved bytes inside "how to compute"; this section saves an entire FFT inside "what to compute" — first ask the caller which slice it actually needs, then talk about how fast to compute that slice. And the precondition for "computing less" is locking the uncertainty inside a verifiable interval: the window is in charge of fast, the interval is in charge of accurate, and when in doubt, fall back.

17. Fusion All the Way: the Frequency-Domain Three-in-One

The second cut returns to the main battlefield of Section 15. Deep fusion folded "data movement" into the load/write-back phases of "compute", but at three-level-decomposition sizes (N > 222) the frequency-domain middle stage still runs as a chain of three launches: a last-level R-point forward FFT (k_stageR_fused, forward direction) → a frequency-domain square or 2-for-1 unpack-multiply (k_pw_sqr / k_pw_2for1) → a first-level R-point inverse FFT. Three kernels, each doing one full pass of global-memory reads and writes over the same batch of data — and at 228 points, one pass is 4GB. Measure the working set once more: R ≤ 64, one CTA covers 32 columns, and an entire R×32 slab is only 64×32 complex numbers ≈ 32KB (even the two slabs of the packed path total just 66KB) — Section 15's fusion criterion lights up green. This time the fusion is no longer "movement + compute" but "compute + compute + compute":

The squaring path falls out naturally; the 2-for-1 multiply path is the hard part: frequency-domain unpacking has to touch the mirror bins at the same time (Section 12's A/B conjugate split), and the mirror bin lands in a different R×32 slab. The fix is to turn the pairing into data: the host pre-generates a table of unique mirror tile-pair descriptors (at the 222 inner size, 65,535 pairs in all); an ordinary CTA claims one pair of descriptors by blockIdx.x, loads both mirror slabs together, transforms each, and cross-multiplies them; the self-conjugate bins and the misaligned first tile (the s=0 neighborhood, column Q/2) go to a dedicated CTA on a SPECIAL branch — the branch tax lands on just 1 block, while the remaining 65,535 blocks take the branch-free fast path. The two slabs plus the transform workspace exceed the 48KB default shared-memory cap, so the launch side requests more with an opt-in attribute:

The two cuts — this one and the middle product — are almost perfectly orthogonal: one trims the number of FFTs (what gets computed), the other speeds up the middle stage of every FFT (how it gets computed), and neither encroaches on the other's gains. This is the second paycheck collected on Section 15's rule — and the criterion has never changed: if the working set fits in shared memory, fuse.

18. The Carry Pipeline: In Passing, Fused, and Sparse

Once multiplication got cheap, what surfaced on the profiler's leaderboard was the carry chain — the "in-segment exact + lookahead scan" kernels that landed in Section 13. The bytes are all coalesced now; only one problem is left: the number of full-array sweeps. This chapter takes two steps: first get other kernels to do the sweeping work "in passing", then make the sweeps themselves disappear.

18.1 strip, done in passing

The first step comes from a simple observation: strip() (find the highest nonzero limb) used to be a standalone full-array sweep kernel — yet the carry-apply kernel k_scan_apply has to see every final digit anyway, so why sweep again? Have it record each chunk's highest nonzero position in passing, and all that remains is reducing a few thousand chunk maxima instead of 30-odd million limbs — the standalone strip sweep is out of the game for good.

Maxim 5: A kernel that already holds the data should finish, in passing, everything that needs that data. In a memory-bound world, a second full-array sweep is a crime against bandwidth — "in passing" (fusion of concerns) and Section 15's "fusion" (fusion of kernels) are two spellings of the same idea.

18.2 Fused and sparse

With the in-passing strip in place, the profile shows that multiply normalization (raw int64 FFT coefficients → valid base-1000 digits) is still a volley of five launches, each kernel doing one full read-and-write pass over 30-odd million limbs: k_carry_seg → k_carry_fix → k_scan_agg → k_scan_chunks → k_scan_apply. The memory-access-shape homework is all done (Section 13); the next layer of gains can only come from running fewer passes. The four-step rework that follows is the capstone project of the in-passing principle (Maxim 5):

Step one: fold the cross-segment fix into k_carry_seg. Phase ② (adding the previous segment's outgoing carry into this segment) used to be a separate kernel that read every digit again. Now adjacent segments within a CTA hand their outgoing carries over directly through shared memory; across CTA boundaries there is no inter-block synchronization — so let thread 0 of each block recompute the previous segment's outgoing carry from the raw input, trading a few dozen extra reads for an entire global kernel:

k_carry_fix disappears from the multiply-normalization path entirely — taken alone this step lands inside the noise band, but it is the foundation for the two steps that follow.

Step two: compute G/P in passing as well. The (g, p) semigroup of the lookahead scan (Section 13: g = a carry is certainly generated, p = a carry would propagate through) used to be computed by k_scan_agg re-reading the digits — yet at that very moment the digits are lying in k_carry_seg's shared memory. So, in passing: during in-segment normalization, record with a bitmask whether each digit equals 999/0, compute each segment's g/p at segment end, and have thread 0 aggregate the whole chunk's G/P. k_scan_agg is dropped from the multiply path altogether; only addition and subtraction still use it. Step three: never compute the same thing twice. Each segment's g/p is only two bits; pack them into one byte in the Tgp table (about 1 MiB of extra peak memory over the whole run), and the downstream apply just looks it up instead of having every thread walk 32 limbs again.

Step four closes the net: the sparse apply. The key observation: after in-segment exact normalization, the vast majority of segments already hold valid digits and see an incoming carry of 0 — yet the generic k_scan_apply, no questions asked, reads every segment in full and writes it back in full. The sparse version processes only the prefix the carry actually touches:

# Excerpt from pi-numba/common.py — sparse apply: write only the carry-touched prefix; the SegHi invariant preserves strip's exactness
@cuda.jit

    i0 = seg * CSEG
    delta = int64(thr_in[cuda.threadIdx.x]) if sign > 0 else -int64(thr_in[cuda.threadIdx.x])
    if seg > 0 and i0 < n:
        delta += int64(res[seg - 1])
    touched = 0
    changed_hi = 0
    k = 0
    while k < CSEG and i0 + k < n and delta != 0:
        v = int64(s[i0 + k]) + delta
        delta = floordiv_b(v)
        d = int32(v - delta * LIMB_B)
        s[i0 + k] = d
        touched = k + 1
        if d != 0:
            changed_hi = k + 1
        k += 1
    old_hi = int(SegHi[seg]) if seg < nseg else 0
    seg_hi = old_hi if old_hi > touched else changed_hi

Note the last two lines: strip's exactness (only just folded into the scan in subsection 18.1) was nearly destroyed by sparsification — without reading the whole segment, you cannot know where the highest nonzero digit sits. The insurance policy is SegHi: while normalizing, k_carry_seg uses a bitmask to record each segment's highest nonzero position in passing; after a sparse modification, if the old high mark falls in the untouched suffix it is still exact, and otherwise the changed_hi of the rewritten prefix supplies it. Every "skip" is backed by an invariant.

Measured on the full 1-hundred-million-digit run: multiply normalization shrinks from 6 kernels (standalone strip + a volley of five full sweeps) to 4; together with the frequency-domain three-in-one, the run's total kernel-launch count falls from 26,563 to 23,251.

"No sweep" beats any "faster sweep" — in the sparse apply, a delta=0 segment doesn't touch a single byte. This "in passing + fused + sparse" combination hardened into the shipping carry pipeline, and it is the entire source of this rung's gain on the ladder.

Maxim 6: The endgame of optimizing a full-array sweep is making the sweep disappear. After local normalization, a carry usually affects only a very short prefix — build a fast path for the common case, keep the generic path for the rare one, and let every "skip" be backed by an invariant (SegHi, the boundary of the range assumption).

20. Closing Remarks

A few things worth taking with you:

• Algorithm before implementation: quadratic convergence compresses 100 million digits into 26 rounds of big operations — pick the right algorithm, and the shape of the problem is already half right.
• Profiling is not a ritual: "the real culprit is not the suspect" played out again and again (σ scattering, the full-array carry sweep, the dispatcher fee) — every conviction came from the profiler and the stopwatch.
• Save the most bytes, not the most instructions: 2-for-1 packing, deep fusion, sparse carry — every major speedup in this story was a bandwidth ledger.
• The deepest saving is to "compute less": the middle-product section alone is worth nearly as much as all the other optimization tiers combined — first ask what the caller actually needs, and only then how to compute it fast; and the license to "compute less" is issued by rigorous intervals and automatic fallback (Section 16), with adversarial tests as its annual inspection (④ in this section).
• The library's kernel boundary is a line you draw: inviting the math library into your kernel buys you not a faster FFT, but sovereignty over the load and write-back phases. That line was first drawn through data movement (Section 15), then through the frequency-domain middle stage (Section 17) and the carry chain (Section 18) — every step of the performance work grew out of that sovereignty.

Gauss–Legendre/Brent–Salamin, FFT big-number multiplication, Newton reciprocal/square root, carry-lookahead, and the middle product are all classical methods of high-precision arithmetic.
