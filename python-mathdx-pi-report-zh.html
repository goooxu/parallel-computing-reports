1. 开篇：这篇教程教什么

在 GPU 上做数学，有两种截然不同的用库姿势。第一种是打电话：调用 cuFFT 这类主机端库，一个函数进去，库替你把事情做完——它自己发射 kernel、自己管调度，你等结果。第二种是把库请进自己的 kernel：cuFFTDx（NVIDIA MathDx 里的设备端 FFT 库）不替你做完任何事，它只是把「一个线程块内的小 FFT」变成你 kernel 代码里的一行普通语句，数据在你自己的寄存器和共享内存里，进出这行语句前后的每一个字节搬运都归你管。

这篇教程完整走一遍第二种姿势：怎么在 Python 里拼出变换的「菜谱」、怎么在一个 CUDA kernel 里执行它、怎么把「一个块装得下」的小变换拼成 228 点的大 FFT，以及——最重要的——「自己掌管数据进出」这件事到底值多少性能。为了让答案有分量，我们选了一个足够狠的贯穿案例：在一块 H100 上，把 π 计算到 100,000,000 位，全部大数运算从零实现。先亮成绩单：

3. 打电话，还是请进厨房

先看熟悉的主机调用模型。用 cuFFT 做一次「变换—逐点乘—逆变换」，主机代码的典型形态是：

# 示意：主机调用模型的一次大数乘法
buf = to_complex(a_limbs)                      # ① 前处理：自己先发射一个 kernel、一遍全长读写
fa = cp.fft.fft(buf)                           # ② 打电话：库发射它自己的 kernel 做正变换
fb = cp.fft.fft(to_complex(b_limbs))           # ③ 第二个操作数，再打一次电话
spec = fa * fb                                 # ④ 逐点乘：cupy 为这一行再发射一个 kernel
raw = cp.fft.ifft(spec)                        # ⑤ 逆变换（再打一次电话）
digits = cp.rint(raw.real)                     # ⑥ 取整：又一遍全长读写
carry_kernel[blocks, 256](digits, out_limbs)   # ⑦ 逐位进位：自己的 Numba kernel 收尾
# 每步之间数据都要整片经过显存；库 kernel 内部一行也插不进去

这个模型的合同很清楚：你在 CPU 上建好 plan（库的执行计划句柄），把数据放进显存，打一个电话，库发射它自己的 kernel 把这一步做完。省心，成熟，快——但有两条写死在合同里的限制：①每一步之间，数据必须整片躺回显存（库 kernel 的输入输出只认显存指针）；②库 kernel 内部你一行代码都插不进去（想在装载时顺手做个类型转换？对不起，先自己发射一个 kernel 把类型转好）。

设备内联模型把这两条合同全部撕掉。cuFFTDx 经 nvmath-python 的设备端 API 进入 Python：你在构造期用「关键字参数拼配料」的方式声明想要什么变换，它给你一个描述符；在你自己的 kernel 里，这个描述符的 execute() 就地完成一个线程块内的 FFT——不发射新 kernel，数据全程在你的寄存器和共享内存里：

4. 运行示例登场：一亿位 π，为什么它能逼出一切

5. 数学积木 I：limb 大数、进位，与 999…9 暗礁

表示。把大整数拆成许多 0～999 的小数字，每个叫一个 limb，整体是一个 int32 数组（低位在前）。选基 B=1000（=10³）是因为最终要输出十进制：每个 limb 恰好 3 位数字，输出零成本。一亿位的 π 就是约 3,333 万个 limb 的数组（130MB 出头）。

加法与进位。小学的「逐位相加、逢千进一」是天生串行的（第 k 位的进位要等第 k−1 位），正撞在第 2 节心法的枪口上。GPU 上改成两步：先整片相加（有些 limb 暂时超过 999，不管它），再并行传播进位。传播这一步藏着全项目最险的暗礁：牛顿迭代（第 7 节）会系统性地制造1 后面跟几十万个 0（或差一点时的 999…9）这种数——一个 +1 的进位要穿过整条 0/9 长链。「多趟传播、每趟推一格」的朴素并行版在这种输入上退化成 O(n) 趟：百万位规模的一次乘法，进位就要传播 166,679 趟——十几万次 kernel 发射，光发射开销就以十秒计，完全不可用。

解法与硬件加法器的经典答案同源——进位前瞻（carry-lookahead）。观察：残余进位只会「穿过」999（借位只会穿过 000），在其他任何数字上都会被吸收。于是「位置 i 会不会收到进位」＝「i 下方最近的非 999 位置是否点火」——这是一个前缀扫描（prefix scan），一次算完所有位置，链再长也一趟到底：

# 串行进位（一次只能走一格）              # 并行前缀版（一次扫描走到底）
carry = 0                                 # 残余进位 ∈ {0,±1} 时：
for i in range(n):                        #   999 会让进位「穿过」(propagate)
    v = s[i] + carry                      #   ≥1000 必然「产生」进位 (generate)
    s[i] = v % 1000                       # (g,p) 是可结合的半群 → 分块前缀扫描
    carry = v // 1000                     # CUDA 三个 kernel：块内聚合→块间串链→应用
教训：向量化原语的「平均情况」性能可能藏着 O(n) 的最坏情况，而数值算法恰恰会系统性地制造最坏情况输入。快算法会把压力转移给基础设施。

减法用对称的借位写法；比较大小从最高位找第一个不同的 limb；「除以 2」有个漂亮的并行技巧：基 1000 是偶数，第 i 位结果 = d[i]/2 + 500·(d[i+1] 是否为奇)，所有 limb 一次算完、无需传播。这些原语的 CUDA 实现全在 common.py。进位的最终形态（两阶段精确进位）在第 13 节。

6. 数学积木 II：FFT 乘法与 0.25 红线

把大数看成「1000 进制多项式」，两数相乘＝系数做卷积。朴素卷积是 O(n²)——对 3,333 万 limb 大约是 1015 次乘加，要算上几十分钟，而我们要做 1062 次。卷积定理给出近路：

x⋅y=F−1(F(x)⊙F(y)), ☉ 为逐点相乘

浮点怎么算出精确整数？FFT 内部用 float64，但 float64 能精确表示到 253 ≈ 9×1015。一亿位时卷积系数最大约 n·999² ≈ 3.34×107×106 ≈ 3.3×1013——离上限还有约 270 倍。但这只是「系数本身装得下」；FFT 的几千万次浮点乘加还会累积舍入误差，所以每次乘法都设一道安全阀：检查每个浮点结果距最近整数的距离，超过 0.25 立即报错退出——绝不悄悄算错：

# 节选自 pi-numba/fft.py —— 每次乘法的 0.25 红线：残差在设备侧累积，检查点回读判罚
def fft_error_read():
    bits = int(g_err.get()[0])
    cp.cuda.runtime.memsetAsync(g_err.data.ptr, 0, 8, 0)
    return np.frombuffer(np.int64(bits).tobytes(), dtype=np.float64)[0]

def fft_error_checkpoint(where=""):
    # 残差在设备侧累积、检查点回读判罚（红线不撤）
    err = fft_error_read()
    if err > g_cnt.fft_err_max:
        g_cnt.fft_err_max = err
    assert err < 0.25, f"FFT rounding error too large: {err} at {where}"

一亿位全程的实测最大误差是 0.00781 ——比红线低一个数量级，但已经能看出：基 1000 在 228 点这个规模上接近极限了（再大十倍就得换基 100）。红线的价值正在于此：它把「精度快不够了」变成显式可观测的量，而不是某天悄悄输出一串错误数字。

很小的乘法（两数长度之积——即 O(n²) 的乘加次数——≤ 1.6×107）直接用 schoolbook kernel，躲开 FFT 的固定开销——牛顿爬升的低层级全是这种小乘法。

7. 数学积木 III：牛顿迭代——只用乘法造出除法与开方

算 π 还需要除法和开方，但我们只想用已经造好的乘法。牛顿迭代做到了。求倒数 1/T：

xn+1=xn⁢(2−T⋅xn)

求倒数平方根 1/√K（开方即 √K = K·(1/√K)）：

yn+1=yn⁢(3−K⋅yn2)2

两个迭代都只有乘法、减法和「除以 2」（第 5 节的并行取半），而且都二次收敛——每迭代一步，正确位数翻倍：15 → 30 → 60 → … → 108，从一个 float64 种子出发约 23 步到位。在整数世界里，1/√K 被表示成 y ≈ Bh/√K 的定点放大形式（h 比 K 的 limb 数略大，实现中取 h = len(K)+2；于是 K·y² ≈ B2h，第 9 节代码里的残差 B2h − K·y² 正来源于此），「除以 B 的幂」恰好是把数组低位整段丢掉——精确且零成本。

记住「每步位数翻倍」——第 8 节选算法、第 9 节的两件武器，全部建立在这一个性质上。

8. 算法主角：Gauss–Legendre 与定点缩放

基础设施齐了，用什么公式算 π？最出名的莱布尼茨级数是反面教材：

π4=1−13+15−17+⋯

一百万项才换来 6～7 位。历史上有收敛快得多的级数（好的每项贡献十几位），但级数有个共同天花板：正确位数与项数成正比（线性收敛），而且海量小项的合并是 GPU 最不喜欢的碎负载。Gauss–Legendre（Brent–Salamin，1976）不做级数求和，而是反复施加一组极简单的变换：

a0=1,b0=12,t0=14,p0=1
an+1=an+bn2, bn+1=anbn, tn+1=tn−pn(an−an+1)2, pn+1=2pn
π≈(an+bn)24tn

a 取算术平均、b 取几何平均，两者以惊人的速度彼此靠近（共同极限叫「算术几何平均数」AGM）。关键性质与牛顿同款：二次收敛，一致位数每轮翻倍——一亿位只要 26 轮，且每轮都是整片数组级别的大运算，正是 GPU 喜欢的负载形状：

代价同样明确：每一轮都要在一亿位精度上开一次平方根（b′=√(ab)），收尾还有一次同精度除法。这个算法能不能快，完全取决于大数乘法有多快、牛顿迭代做得聪明不聪明——正好把压力全部传导给我们的基础设施。

定点缩放解决「实数进整数库」：所有量一律乘 10p 存成整数（p = 100,000,050，3 的倍数 → 恰好 33,333,350 个 limb）。每个实数运算都严格对应一个整数运算：

实数世界定点整数世界（一切 ×10p）用到的原语
(a+b)/2(A+B)/2加法 + 偶基取半（无进位传播）
√(a·b)isqrt(A·B)：A·B 带 102p，开方后恰好回到 10p乘法 + 牛顿开方
t − p·(a−a′)²T − pₙ·(D² 右移 p 位)乘法 + 移位（丢低位=截断）

9. 牛顿的两件武器：精度爬升与热启动

牛顿开方如果只做一次，怎么写都行；但 AGM 每轮都来，加上初始化共 27 次。如果每步牛顿都在全精度上做乘法，一次开方 ≈ 23 步全尺寸大乘法，27 次开方 ≈ 600 多步——足以吞掉换算法省下的一切。两件标准武器：

9.1 精度爬升（ramping）

牛顿第 k 步只有约 2k 位是正确的——那就别在全精度上白算！把精度分成层级（每层 limb 数翻倍），最低层用 Python 大整数（math.isqrt）精确播种，然后逐层上爬：每层只取 K 的最高 m 个 limb 参与运算，做一步牛顿恰好把正确位数从「半层」推到「全层」。因为每层运算量是上一层的两倍，总代价 ≈ 最后一层的 2 倍——一次开方 ≈ 常数次全尺寸乘法。

9.2 热启动（warm start）

相邻两轮的 K = aₙbₙ 高位几乎一样（第 n 轮已一致 ~2n 位），而牛顿的中间量 y ≈ 1/√K 我们本来就留着——把上一轮的 y 直接当种子，跳过大部分爬升层级。最后一层照例迭代到修正量为零，天然自我纠错：就算热启动过于乐观，也只是多做一两步，不会算错。还有一个小而准的收尾优化：当修正量长度不足 y 的一半时，二次收敛保证下一步修正必然小于 1 ulp——确认步可以直接免掉。这一条把全程牛顿步数从 479 压到 443，大乘法从 1145 次压到 1062 次：

10. 一个线程块里的 2048 点 FFT

从这一节起，主角登场。先把第 3 节尝过一口的「菜谱」正式端上桌：cuFFTDx 的一切配置——变换多大、什么精度、正向还是逆向、每块做几批、在块级执行——在 nvmath-python 里就是一份关键字参数表，一次普通的 Python 函数调用拼成一个 FFT 描述符。全项目所有 FFT 菜谱都从同一座小工厂出锅：

# 节选自 pi-numba/fft.py —— 「菜谱」工厂：一次函数调用拼配料，返回 FFT 描述符
from nvmath.device import FFT as DxFFT

def _dx(size, inv, fpb, ept=None):
    kw = dict(fft_type="c2c", size=size, precision=np.float64,   # 配料：复数变换、多大、fp64
              direction=("inverse" if inv else "forward"),        # 正向 / 逆向
              ffts_per_block=fpb, execution="Block")              # 每块几批、块级执行
    if ept is not None:
        kw["elements_per_thread"] = ept                           # 不指定就让库自选最优值
    return DxFFT(**kw)                                            # ← 一张菜谱，不发射任何 kernel

和「先建 plan 再打电话」的旧模型最大的不同：这个描述符不替你做任何事——它是一张菜谱。构造发生在 Python 运行期，但拿到手的一刻，菜谱就把「厨房该怎么开」全部告诉你：storage_size 是每线程要备几格寄存器、elements_per_thread/stride 是装载循环的趟数与交错步幅、value_type 是寄存器与共享内存数组的元素类型、block_dim/shared_memory_size 是发射时的块形与工作区大小。这些 traits 会以 Python 常量的身份被闭包捕获进 CUDA kernel——Numba 编译那一刻全部折叠成常量，生成的机器码与把数字写死没有两样。「菜谱在先、厨房照单」的思想一点没变，只是拼菜谱的动作从此发生在构造期：

于是 kernel 的正确写法是一座工厂函数：先构造描述符、读出 traits，再在函数体内定义 CUDA kernel，让闭包把这些常量焊进去。下面是本项目 P 级的 kernel B——每个线程块负责一行 2048 点 fp64 复数 FFT，装载/写回相位顺手乘上四步法的外层 twiddle（为什么要乘、乘的是什么角度，第 11 节展开）：

# 节选自 pi-numba/fft.py —— kernel B（P 级）：每块一行 2048 点 FFT ＋ 外层 twiddle
def make_stageB(inv):
    fft = _dx(FFT_P, inv, 1)                    # FFT_P = 2048，每块 1 批
    st, ept, stride = fft.storage_size, fft.elements_per_thread, fft.stride
    vt = fft.value_type
    IS_INV = inv                                # ↑ 读出 traits：闭包捕获，编译时折叠成常量

    @cuda.jit
    def k_stageB(data, N, rowStride):
        rd = cuda.local.array(shape=(st,), dtype=vt)   # 数据住在你的寄存器里
        sm = cuda.shared.array(shape=(0,), dtype=vt)   # 动态共享内存：库的工作区
        j = cuda.blockIdx.x                            # 本块负责第 j 行
        base = cuda.blockIdx.y * rowStride + j * FFT_P
        invN2 = 2.0 / N
        for i in range(ept):                           # 装载相位：你写的循环
            e = cuda.threadIdx.x + i * stride
            if e < FFT_P:
                v = data[base + e]
                if IS_INV:
                    sn, cs = libdevice.sincospi((j * e) * invN2)
                    v = v * complex(cs, sn)            # 逆变换：装载时顺手乘 twiddle
                rd[i] = v
        fft.execute(rd, sm)                            # ← 库在这儿：一行完成 2048 点 FFT
        for i in range(ept):                           # 写回相位：也是你写的循环
            e = cuda.threadIdx.x + i * stride
            if e < FFT_P:
                v = rd[i]
                if not IS_INV:
                    sn, cs = libdevice.sincospi(-(j * e) * invN2)
                    v = v * complex(cs, sn)            # 正变换：写回时顺手乘 twiddle
                data[base + e] = v

    return k_stageB, fft                               # kernel 与菜谱一起交给发射方

几个值得停下来看的细节：装载的形状是库定的登记表——每线程管 ept 个元素、彼此以 stride 交错（e = threadIdx.x + i·stride），装载与写回照抄这张表，execute 才认账（if e < FFT_P 是通用护栏，防某些配置下 ept·stride 越过行尾）；fft.execute(rd, sm) 一行——编进你 kernel 里的一次块内设备端调用，不发射新 kernel，2048 点蝶形借共享内存就地完成，结果仍回寄存器 rd；方向分叉在工厂里就消失了——IS_INV 是闭包常量，正、逆两个 kernel 各自编译，机器码里没有一条运行期分支：

twiddle 精度也必须较真：所有相位的分母都是 2 的幂——invN2 = 2.0/N 在二进制里是精确值，j·e 是不超过 222 的整数，乘积一位不丢；libdevice.sincospi 自带精确 π 归约，一次调用同时给出正弦与余弦（Python 侧直接解包成 sn, cs，再用 v * complex(cs, sn) 完成旋转），每个 twiddle 误差 ≤1 ulp——不查表、不累积、现场算。这份「便宜的严谨」正是第 6 节 0.25 红线预算表里最省心的一项。

最后一步，菜谱反过来告诉你厨房开多大——发射参数一个字都不用自己算，照单全收：

def run_stageB(buf, inv, Q, N, R=1, rowStride=0):

「描述符工厂 + traits 折叠 + 缓存」这个三件套是全篇 FFT 侧的地基：同一个 _dx 之后还要为 Q 点行 FFT（kernel D）、R 点外层 FFT 乃至第 17 节「正逆两份描述符同锅」的频域三合一反复出锅，_get/_kcache 保证每种配置只编译一次——「一次编译、处处发射」在第 14 节结算 Python 发射税时是一句关键证词；拼出来的东西对不对，则由第 19 节的验证梯子把关。单块的天花板依旧存在：fp64 复数一个线程块只装得下几千点，而一亿位 π 需要 228——下一节就把这块 2048 点的积木拼成大厦。

11. 超出一个块：四步法、三级分解，与从不重排的频谱

cuFFTDx 的天花板是「一个线程块装得下」：fp64 复数几千点。而一亿位 π 的大乘法需要 N = 228。经典解法是四步法：把长度 N = P·Q 的数组看成 P×Q 矩阵（n = Q·i + j），DFT 精确拆成两轮小 FFT 夹一个逐点相位：

X[ki+Pkj]=∑jWQjkj⋅WNjki⋅(∑ix[Qi+j]WPiki)

N 再大，就把「内层 222」当成新的积木再拆一层——三级分解 N = 222 × R（R ≤ 64），覆盖到 228：

这里有个省了大功夫的设计：kernel B 输出的频率顺序天然是「置乱」的（数位反转的混合序）。教科书做法是再加一趟全量重排恢复自然序——但我们的 FFT 只服务于卷积：正变换后马上逐点乘、马上逆变换。逐点乘不看座位号：只要正逆变换用同一套置乱序 σ，两个谱逐元素对齐即可，谁坐哪个位置根本无所谓——就像两批观众按同一张乱序座位表入座，配对关系一点不乱。于是重排一趟都不用做，逆变换就是正变换的严格镜像。唯一需要「查座位表」的场合是 2-for-1 解包（第 12 节）要找共轭伙伴 σ(N−k)，为此备一对便宜的映射函数：

# 节选自 pi-numba/fft.py —— 置乱谱的「座位表」：存储序 σ ↔ 真实频率 k 的双向映射
@cuda.jit(device=True)
def ki_of_s(s, Qin):
    pos = s // Qin
    kq = s - pos * Qin
    return pos + 2048 * kq

@cuda.jit(device=True)
def sigma_of_k(k, M, R, Qin):
    kb = k // M
    ka = k - kb * M
    kq = ka >> 11
    r = ka & 2047
    return (r * Qin + kq) * R + kb

13. 少发射、合并访问：两阶段进位与访存形态

13.1 kernel 发射也要钱

13.2 让每一个字节都合并

发射的账记到台面上之后，profile 的焦点回到访存形态：同样的字节数，合并访问（相邻线程读相邻地址）与散射访问的带宽能差几倍。此时的头号热点正是 2-for-1 的解包 kernel——它按频率 k 主序迭代、按座位号 σ 散射访问，一度独占近三分之一的运行时间。把迭代变量换成 σ 本身（k 反查改用便宜的正向映射 ki_of_s），自身访问变成纯连续、共轭伙伴一侧也呈镜像连续段——榜首热点应声退位。配上扫描装载的共享内存整块化、T − pₙ·(d²≫p) 三合一 kernel 等零碎收尾。

格言二：发射数和访存形态是「整片数组」心法的另一半。把循环改成整片运算只是开始；整片运算发射多少次、每个线程摸到的地址挨不挨着，决定它跑多快。

14. Python 的发射税

14.1 dispatcher：每次发射 40 µs 的海关

上一章数的是发射的次数，这一章数发射的单价。Numba 的 kernel 对象是一个 dispatcher：每次调用都要 configure 一遍网格参数、在 C 层做一遍类型匹配、再对每个 cupy 数组参数解析一次 __cuda_array_interface__，合计每发射约 40 µs——这是解释器世界特有的一道海关，收在 GPU 侧启动延迟之外。一次 1 亿位运行要发射 23,251 次 kernel，税单加总约 1 秒量级：kernel 没变、发射数也几乎没变（图 11），钱全花在 Python↔驱动的边境上。

# 节选自 pi-numba/common.py —— 薄发射：预 specialize + overloads 直连，
# 跳过 dispatcher 每次调用的 configure/typeof
def prime(kern, *args):
    """惰性 specialize + 缓存；返回可直接 .launch 的已编译 _Kernel。"""
    krn = _spec_cache.get(kern)
    if krn is None:
        spec = kern.specialize(*args)
        krn = next(iter(spec.overloads.values()))
        _spec_cache[kern] = krn
    return krn

def launch(kern, grid, block, smem, *args):
    g_cnt.launches += 1
    args = tuple(_nb_twins.get(id(a), a) for a in args)   # 常驻缓冲 → numba 化身
    prime(kern, *args).launch(args, _d3(grid), _d3(block), core_stream0(), smem)

对策不是绕开 Numba，而是让它的慢路径只走一次：prime() 首次调用时 specialize 并缓存已编译的 _Kernel，launch() 此后直连它的 .launch 入口，configure、类型匹配、参数解析三道手续整体出局。前提写在源码注释里：每个 kernel 的调用签名全程稳定（arena 视图只有 dtype/ndim 进类型，标量侧无 int/float 分叉），selftest 全覆盖兜底。

14.2 常驻化身与 pinned 回读

直连之后，发射路径上最贵的剩余工序是给每个 cupy 参数解析 CAI——而对常驻缓冲（FFT 工作区、g_pos 这些整场不换的对象），这份解析每次都得出同样的结果。register_twin() 把它们一次性包成 numba device array，launch() 里按 id 换成化身，走 is_device_memory 的快分支，CAI 解析归零；同一味药也治主机回读：read_pos() 用预分配的 pinned 缓冲做异步拷贝，免掉每次回读的临时分配。

14.3 一个负结果与 JIT 预热：测量胜于推理

common.py 文件头还有两行必须先于 import numba 注入的环境变量：NUMBA_CUDA_ARRAY_INTERFACE_SYNC=0 与 NUMBA_CUDA_LOW_OCCUPANCY_WARNINGS=0（setdefault 写法，外部显式置 1 可回退）。前者曾被寄予厚望——全程单默认流、同流天然有序，per-launch 的 CAI 流同步纯属冗余，理论上每次发射都能省一笔——A/B 实测增益 ≈0。负结果照样进正文：它证明真正的肥肉在 dispatcher 本身而不在这次同步，也把「测量胜于推理」又演了一遍——假说负责排优先级，秒表负责定生死。

最后一笔税只交一次：同进程第一遍运行含全部 kernel 的 JIT 编译，墙钟 177.5 秒；其后热跑只要 ~5.630 秒。生产姿势因此固定为「1 次预热 + N 次正式」，本文所有计时都取热跑。

格言三：解释器的世界里，每一次跨界（Python↔驱动）都要计价——把跨界摊薄，而不是幻想消灭跨界。specialize 缓存一次、化身注册一次、pinned 缓冲留一块，都是同一个动作：把 O(发射数) 的每次手续，折旧成 O(1) 的一次性投资。发射数几乎没动（图 11），动的是单价。

15. 深度融合：设备端库的独门优势

至此，每次大乘法的中段仍是一条十几个 kernel 的链：纯搬运 kernel（转置/置换/装载/取整）与计算 kernel（FFT/逐点乘）交替出现，每个都是一遍全量显存读写——228 点时一遍就是 4GB。现在兑现第 3 节的伏笔：既然 FFT 是你 kernel 里的一行，它前后的装载/写回相位就是你的代码——把相邻的搬运 kernel 折进计算 kernel 的装载/写回相位，发射和整遍带宽一起省。这样的动词在主机调用模型里根本不存在（那里连「装载时顺手转个类型」都要走回调的窄门），而在设备端模型里只是改自己 kernel 里的一个循环。判断能不能融只有一条法则：融合后一个线程块的工作集必须装进共享内存。逐一排查：

接下来的三节会证明：同一套方法论——先 profile、恒等式安全网、严格的 A/B 纪律（第 19 节）——还能把性能继续提升：先用精确 middle product 砍掉四分之三座 FFT（第 16 节，一刀 −26.1%），再让进位流水线顺手化、融合化、稀疏化（第 18 节），把频域中段融成一个 kernel（第 17 节）；连同第 14 节的发射税治理与共享内存 padding 这样的零碎。

两刀与 middle product 几乎完全正交：一刀砍 FFT 的座数（算什么），一刀提速每座 FFT 的中段（怎么算），互不侵占对方的收益。这是第 15 节那条法则的第二次领工资——判据没变过：工作集装得进共享内存，就融。

18. 进位流水线：顺手、融合与稀疏

乘法便宜下来之后，profiler 榜单上浮出水面的是进位链——第 13 节落地的那套「段内精确 + 前瞻扫描」kernel。字节都合并了，剩下的问题只有一个：全量扫描的遍数。这一章两步走：先让别的 kernel「顺手」把扫描的活干了，再让扫描本身消失。

18.1 顺手做完的 strip

第一步来自一个简单的观察：strip()（找最高非零 limb）此前是一个独立的全数组扫描 kernel，可进位应用 k_scan_apply 本来就要看到每一个最终 digit，何必再扫一遍？让它顺手记下每个 chunk 的最高非零位置，之后只需归约几千个 chunk 极值，而不是三千多万个 limb——独立的 strip 扫描从此整个出局。

格言五：已经把数据拿在手里的 kernel，应该顺手做完所有需要这份数据的事。访存主导的世界里，第二次全数组扫描是对带宽的犯罪——「顺手」（fusion of concerns）和第 15 节的「融合」（fusion of kernels）是同一个道理的两种写法。

18.2 融合与稀疏

顺手的 strip 落地之后再看 profile，乘法归一化（FFT 的 int64 原始系数 → 合法 base-1000 digit）的路径还是一条五连发射，每个 kernel 都对三千多万 limb 做一遍全量读写：k_carry_seg → k_carry_fix → k_scan_agg → k_scan_chunks → k_scan_apply。访存形态的功课都做完了（第 13 节），下一层收益只能来自「少跑几遍」。接下来的四步改造，是「顺手原则」（格言五）的毕业设计：

第一步：段间修正折进 k_carry_seg。原本阶段②（前段出位加进本段）是独立 kernel，重新把 digit 读一遍。现在 CTA 内部经共享内存直接传递相邻段的出位；CTA 边界没有跨块同步——让每块的 0 号线程从原始输入重算前一段的出位，几十次额外读取换掉一整个全局 kernel：

k_carry_fix 从乘法归一化的路径上整个消失——这一步单独看落在噪声带里，但它是后两步的地基。

第二步：G/P 也顺手算。前瞻扫描的 (g, p) 半群（第 13 节：g=必然产生进位，p=进位会穿过）原本由 k_scan_agg 重新读一遍 digit 来算——可 digit 此刻就躺在 k_carry_seg 的共享内存里。顺手：段内归一化时用位掩码记录「digit 是否等于 999/0」，段尾算出每段 g/p，0 号线程再聚合出整个 chunk 的 G/P。乘法路径上的 k_scan_agg 全部免掉，只剩加减法在用。第三步：算过的不再算第二遍。每段 g/p 只有两个 bit，打包成一字节存进 Tgp 表（全程峰值多花约 1 MiB），下游 apply 直接查表，免去每线程再遍历 32 个 limb。

第四步才是收网：稀疏 apply。关键观察：乘法系数经过段内精确归一化后，绝大多数段已经是合法 digit、入口进位为 0——而通用 k_scan_apply 不管三七二十一，把每个段完整读一遍再完整写一遍。稀疏版只处理进位实际触及的前缀：

# 节选自 pi-numba/common.py —— 稀疏 apply：只写进位触及的前缀，SegHi 不变量保住 strip 精确性
@cuda.jit
def k_scan_apply_sparse(s, n, sign, res, Cin, Tgp, SegHi, Hc):
    ...
    i0 = seg * CSEG
    delta = int64(thr_in[cuda.threadIdx.x]) if sign > 0 else -int64(thr_in[cuda.threadIdx.x])
    if seg > 0 and i0 < n:
        delta += int64(res[seg - 1])
    touched = 0
    changed_hi = 0
    k = 0
    while k < CSEG and i0 + k < n and delta != 0:
        v = int64(s[i0 + k]) + delta
        delta = floordiv_b(v)
        d = int32(v - delta * LIMB_B)
        s[i0 + k] = d
        touched = k + 1
        if d != 0:
            changed_hi = k + 1
        k += 1
    old_hi = int(SegHi[seg]) if seg < nseg else 0
    seg_hi = old_hi if old_hi > touched else changed_hi

注意最后两行：strip 的精确性（18.1 节刚并进扫描）差点被稀疏化毁掉——不读整段就不知道最高非零位在哪。保险单是 SegHi：k_carry_seg 归一化时顺手用位掩码记下每段最高非零位置；稀疏修改后，若旧最高位落在未触及的后缀里它仍然精确，否则由被改前缀的 changed_hi 给出。每一次「跳过」都有不变量作保。

实测（1 亿位全程）：乘法归一化从 6 个 kernel（独立 strip + 五连全量扫描）压到 4 个；与频域三合一合计，全程 kernel 发射数从 26,563 降到 23,251。

「不扫描」赢过任何「更快的扫描」——稀疏 apply 里 delta=0 的段一个字节都不碰。这份「顺手 + 融合 + 稀疏」的组合固化为交付版的进位流水线，也是阶梯上这一档收益的全部来源。

格言六：优化全量扫描的尽头，是让扫描消失。局部归一化之后，进位通常只影响很短的前缀——为常见情形建快路径，为罕见情形留通用路径，并让每一次「跳过」都有不变量（SegHi、值域假设的边界）作保。

手搭 FFT 最怕「悄悄算错」，而阶梯后段的改造——尤其 middle product 和稀疏进位这种「少算」型优化——把风险又抬高了一档。

20. 结语

几件值得带走的事：

• 算法先于实现：二次收敛把一亿位压成 26 轮大运算——选对算法，问题的形状就先对了一半。
• profile 不是仪式：「真凶不是嫌疑人」一再上演（σ 散射、进位全量扫描、dispatcher 手续费），全靠 profiler 与秒表定罪。
• 省最多的字节，而不是最多的指令：2-for-1、深度融合、稀疏进位——历次大幅加速全是带宽账。
• 最深的省法是「少算」：middle product 一节几乎抵得上其余各档的总和——先问调用者要什么，再问怎么算快；而「少算」的执照由严格区间与自动回退签发（第 16 节），由对抗测试年检（本节 ④）。
• 库的 kernel 边界是你画的线：把数学库请进 kernel，买到的不是更快的 FFT，而是对装载与写回相位的支配权。这条线先画进了搬运（第 15 节），又画进了频域中段（第 17 节）与进位链（第 18 节）——性能优化的每一步，都长在这份支配权上。

Gauss–Legendre/Brent–Salamin、FFT 大数乘法、牛顿倒数/开方、进位前瞻、middle product 均为高精度计算的经典方法。
