2. 挑战：为什么不能直接算

浮点数装不下。C++ 的 double、Python 的 float 都是 64 位浮点数，只有约 15～16 位有效数字。要 100 万位，差了 5 个数量级——这不是精度不够，而是类型里根本没那么多位置。

GPU 上没有「大整数」。Python 内置 int 虽是任意精度，但它在 CPU 上单线程运行；而 GPU 的数组库（如 CuPy——可以理解为「GPU 上的数组计算库」，数组默认就在显存里）只有int64、float64 这类定长数组。所以我们要自己用一大片小数字（数组）拼出一个大数，把大数的加减乘除翻译成对整片数组的运算，交给 GPU 上成千上万个核心并行完成。

3. 并行的心法：从「循环」到「整片数组」

CPU＝几位博士生
核心少但强，擅长「一步依赖上一步」的复杂逻辑；跑 for 循环是一次一格。
GPU＝上万名小学生
GPU 有数万个 CUDA 核心，每个只会简单算术，但能同时开工。

要用上 GPU，就得把「逐元素的循环」改写成「对整片数组的一次运算」：

全文最重要的一句话：后面每一个大数运算，我们都会把它写成「对整片 limb 数组的运算」，这样它才真正跑在 GPU 上。做不到这一点的运算（比如天生串行的逐位进位），就要找到它的并行等价形式——这正是第 10 节那场性能事故的主题。

4. 用 limb 数组表示大整数；加法与进位

表示。把大整数拆成许多 0～999 的小数字，每个叫一个 limb，整体是一个 int64 数组（低位在前）。选基 B=1000（=10³）是因为最终要输出十进制：每个 limb 恰好 3 位数字，输出零成本。100 万位的 π 就是约 33.4 万个 limb 的数组。

def from_int(n):                       # Python 整数 -> limb 数组（只用于常数/种子）
    limbs = []
    while n > 0:
        limbs.append(n % 1000)         # 最低 3 位十进制作为一个 limb
        n //= 1000
    return cp.array(limbs, dtype=cp.int64)

def to_decimal_str(x):                 # limb 数组 -> 十进制字符串：每 limb 补足 3 位再拼接
    return "".join("%03d" % int(d) for d in reversed(x.tolist())).lstrip("0")

加法与进位。小学的「逐位相加、逢千进一」是天生串行的（第 k 位的进位要等第 k−1 位）。GPU 上改成两步：先整片相加（有些 limb 暂时超过 999，不管它），再整片传播进位——每趟用一次整片除法算出所有 limb 的溢出 q = c // 1000，把它整体挪高一格加回去。溢出每趟缩小 1000 倍，几趟之后每个 limb 都落回 0～999 附近，只剩下 0/1 的「残余进位」。这些残余怎么一次收干净（而不是再一格格地挪），是第 10 节的主角——先按下不表：

def add(x, y):
    n = max(x.shape[0], y.shape[0])
    x = cp.pad(x, (0, n - x.shape[0]))     # 补齐等长
    y = cp.pad(y, (0, n - y.shape[0]))
    return normalize(x + y)                # 整片相加，进位交给 normalize

def normalize(c):                          # 有界趟：每趟把溢出整体上移一格
    c = cp.pad(c, (0, 8))
    while True:
        q = c // 1000                      # 一次算出所有 limb 的溢出
        if 全为零: return strip(c)
        c = c - q * 1000 + cp.pad(q[:-1], (1, 0))   # 溢出整体挪高一位加回
        if q 只剩 0/1: return strip(_carry_scan(c, +1))   # 残余进位：一次前缀扫描收尾（§10）

减法用对称的「借位」写法；比较大小 cmp 从最高位找第一个不同的 limb；「除以 2」有个漂亮的并行技巧：基 1000 是偶数，第 i 位的结果 = d[i]//2 + 500·(d[i+1] 的奇偶)，所有 limb 一次算完、无需传播——牛顿迭代里会反复用到它。

5. FFT 大整数乘法

原理。把大数看成「1000 进制多项式」，两数相乘＝系数做卷积。朴素卷积是 O(n²)——对 33 万 limb 要约 1011 次乘加，太慢。卷积定理给出近路：在「频域」里，卷积＝逐点相乘。于是 FFT 变到频域、逐点乘、逆 FFT 变回来，总代价降到 O(n log n)：

def mul_fft(x, y):
    m = nx + ny - 1
    s = 2 的幂 ≥ m
    xf = cp.fft.rfft(x.astype(cp.float64), n=s)      # ① 变到频域
    yf = cp.fft.rfft(y.astype(cp.float64), n=s)
    z = cp.fft.irfft(xf * yf, n=s)[:m]               # ② 逐点乘  ③ 变回来
    zr = cp.rint(z)                                  # 四舍五入还原成整数
    err = float(cp.abs(z - zr).max())
    assert err < 0.25, "FFT rounding error too large" # 安全阀：绝不悄悄算错
    return normalize(zr.astype(cp.int64))

浮点怎么算出精确整数？FFT 内部用 float64，但 float64 能精确表示到 253 ≈ 9×1015，而卷积系数最大约 n·999² ≈ 6.7×1011（n 取我们最大的乘法规模），还有一万多倍余量，四舍五入就能无误差还原。更关键的是那行安全阀：每次乘法都检查浮点结果离最近整数有多远，一旦超过 0.25 立即报错——绝不悄悄算错。本次两个实现全程的实测最大误差是 1.5×10⁻⁴ 与 2.1×10⁻⁴，离红线有三个数量级。

很小的数字则用直接的「外积求和」小乘法（schoolbook），避免 FFT 的固定启动开销。

6. 牛顿迭代：只靠乘法做出除法与开方

算 π 还需要除法和开方，但我们只想用已经造好的乘法。牛顿迭代做到了。求倒数 1/T：

x n + 1 = x n ⁢ ( 2 − T ⋅ x n )

求倒数平方根 1/√K（开方即 √K = K·(1/√K)）：

y n + 1 = y n ⁢ ( 3 − K ⋅ y n 2 ) 2

两个迭代都只有乘法、减法和「除以 2」（上一节的并行取半），而且都二次收敛——每迭代一步，正确位数翻倍：15 → 30 → 60 → … → 1,000,000，从一个 double 种子出发约 17 步到位。在整数世界里，「1/√K」被表示成 y ≈ B^(2h)/… 这样的定点放大形式，「除以 B 的幂」恰好是把数组低位整段丢掉，精确且零成本。

记住「每步位数翻倍」这个性质——第 9 节的两项优化（精度爬升、热启动）全部建立在它上面。

7. 选算法：收敛速度决定一切

基础设施齐了，用什么公式算 π？

7.1 级数路线：线性收敛

π 4 = 1 − 1 3 + 1 5 − 1 7 + ⋯

最出名的莱布尼茨级数优美，但 100 万项才换来 6～7 位——纯属反面教材。历史上出现过许多收敛快得多的级数（好的级数每项能贡献十几位），但级数有个共同的天花板：正确位数与项数成正比——这叫线性收敛。要 100 万位，就得老老实实合并几万到上百万个项，而且这些项的合并是海量的「小运算」，恰恰是 GPU 最不喜欢的负载形状（第 3 节的心法反过来说：零碎的小操作喂不饱上万个核心）。

7.2 Gauss–Legendre：每轮位数翻倍

Gauss–Legendre（Brent–Salamin，1976）不做级数求和，而是反复施加一组极简单的变换：

a 0 = 1 , b 0 = 1 2 , t 0 = 1 4 , p 0 = 1
a n + 1 = a n + b n 2 ,  b n + 1 = a n b n ,  t n + 1 = t n − p n ( a n − a n + 1 ) 2 ,  p n + 1 = 2 p n
π ≈ ( a n + b n ) 2 4 t n

a 和 b 一个取算术平均、一个取几何平均，两者会以惊人的速度彼此靠近（这个共同极限叫「算术几何平均数」AGM）。关键性质是二次收敛：a 与 b 一致的位数每轮翻一倍——和牛顿迭代同款。这不是理论上的漂亮话，下面是实测——第 n 轮结束后 a、b 一致的十进制位数：

100 万位只要 19 轮，而且每一轮都是「整片数组」级别的大运算——正是 GPU 喜欢的负载形状。但天下没有免费的午餐：级数路线几乎只用乘加，而 Gauss–Legendre 每一轮都要在 100 万位的精度上开一次平方根（b′ = √(ab)），最后还要做一次同样精度的除法。这个算法能不能跑得快，完全取决于两件事：大数乘法有多快，以及牛顿迭代做得聪明不聪明。

终止判据直接来自二次收敛：设已一致 g 位，下一轮就会一致约 2g 位。当 2g ≥ p + 30（p 是工作精度）时，直接用当前的 a、b、t 出 π 即可，误差压在保护位以下。主循环里就一行：if 2 * agree >= p + 30: break。

8. 定点缩放：让实数进整数库

Gauss–Legendre 的变量是实数（1/√2 = 0.7071…），而我们的库只有非负大整数。办法是定点缩放：所有量一律乘上 10p 存成整数，p = 1,000,050（100 万位 + 50 位保护位，且是 3 的倍数，恰好 333,350 个 limb）。于是每个实数运算都严格对应一个整数运算：

实数世界 定点整数世界（一切 ×10p） 用到的原语
(a+b)/2 (A+B)/2 加法 + 偶基取半（无进位传播）
√(a·b) isqrt(A·B)：A·B 带 102p，开方后恰好回到 10p 乘法 + 牛顿开方
t − p·(a−a′)² T − pₙ·(D² 右移 p 位) 乘法 + 移位（丢低位=截断）
(a+b)²/(4t) (A+B)²·recip(4T) 再右移 乘法 + 牛顿倒数

截断带来的误差每轮不超过几个「最后一位」，乘上 pₙ ≤ 218 也只污染最低六七位，19 轮累计远小于 50 位保护位——这一点在 300 / 3 万 / 30 万位规模上都与基准逐位比对过。基是 10³ 也让「乘除 10 的幂」都变成整段移 limb，精确且零成本。

9. 牛顿迭代的两件武器：精度爬升与热启动

第 6 节的牛顿开方如果只算一次，怎么写都行；但 Gauss–Legendre 每轮都来，共 20 次。如果每一步牛顿都在全精度上做乘法，一次开方要做约 19 步全尺寸大乘法，20 次开方就是约 400 步——足以吞掉换算法省下的一切。这里需要两个标准武器：

9.1 精度爬升（ramping）

牛顿第 k 步只有约 2k 位是正确的——那就别在全精度上白算！把精度分成层级（每层 limb 数翻倍），低层用几十位的 Python 整数精确播种，然后逐层上爬：每层只取 K 的最高 m 个 limb 参与运算，做一步牛顿恰好把正确位数从「半层」推到「全层」。因为每层运算量是上一层的两倍，总代价 ≈ 最后一层的 2 倍——一次开方 ≈ 常数次全尺寸乘法。

9.2 热启动（warm start）

相邻两轮的 K = aₙbₙ 高位几乎一样（第 n 轮已一致 ~2ⁿ 位）。而牛顿迭代的中间量 y ≈ 1/√K 我们本来就留着——把上一轮的 y 直接当种子，就能跳过大部分爬升层级，从「已知一半位数正确」的高层直接进入。最后一层照例迭代到「修正量为零」，天然自我纠错：就算热启动的估计过于乐观，也只是多做一两步，不会算错。

def rsqrt_ramp(bg, K, warm=None, warm_digits=0):
    # y ≈ B^(dk+2)/√K 的牛顿迭代：精度逐层翻倍，只在最后一两层为全精度
    levels = _levels(dk, parity_anchor=dk)     # 例如 [11, 21, 41, ..., 333350, 666700]
    if warm is not None and warm_digits >= 36:
        start = 满足「层级 ≤ 2×可信 limb 数」的最大层级    # 热启动：直接跳进高层
        y = bg.shr_limbs(warm, (dk - levels[start]) // 2)  # 上一轮的 y 降尺度作种子
    else:
        y = 用 Python 整数在最低层级精确播种（十几位）

    for m in levels[start:]:
        Km  = bg.shr_limbs(K, dk - m)          # 只取 K 的最高 m 个 limb ——「工作精度」
        # 牛顿步：y ← y + y·(B^{2h} − K·y²)/2B^{2h}；中间层一步够，最后一层迭代到修正量为 0
        Ky2   = bg.mul(Km, bg.mul(y, y))
        rr    = |B^{2h} − Ky2|
        delta = bg.half(bg.shr_limbs(bg.mul(y, rr), 2 * h))
        y     = y ± delta
    return bg.shr_limbs(bg.mul(K, y), h), y    # √K = K·(1/√K)；y 留给下一轮热启动

10. 一条性能暗礁：999…9 进位长链

这是本项目调优过程中最有戏剧性的一幕，值得单独一节。第一版程序算出了完全正确的 100 万位，但 CuPy 版花了 86.7 秒，其中「最后一次除法」独占 71.4 秒——一次除法比 19 轮迭代加起来还慢好几倍，明显不对劲。逐层计时定位后，罪魁祸首令人意外：进位。

第 4 节的「多趟传播」对随机数据 3~5 趟就干净了。但牛顿迭代天天制造一种特殊的数：T·w ≈ Bg，也就是1 后面跟着几十万个 0（或差一点时的 99999…）。这种数在卷积系数归一化时，一个 +1 的进位要穿过整条 0/9 长链，而「多趟传播」每趟只能把它送过一格——于是一次乘法的进位要传 166,679 趟：

level 333351: mul=34.876s (166679 趟进位!)   ← 修复前的实测（CuPy，单次乘法）

这正是硬件加法器里的经典问题（行波进位 vs 超前进位），解法也是同一个：进位前瞻。先用两三趟有界传播把每个 limb 压到只剩 0/±1 的残余进位，然后观察：进位链只会「穿过」999（或借位链穿过 000），在其他任何数字上都会被吸收。于是「位置 i 会不会收到进位」＝「i 下方最近的非 999 位置是否点火」——这是一个前缀扫描（prefix scan），一次算完所有位置，链再长也一趟到底： 

# 串行进位（一次只能走一格）              # 并行前缀版（一次扫描走到底）
carry = 0                                 def _carry_scan(s, sign):        # s 的残余进位 ∈ {0,1}
for i in range(n):                            prop = (s == 999)            # 999：进位会「穿过」它
    v = s[i] + carry                          gen  = (s >= 1000)           # 这里必然产生进位
    s[i] = v % 1000                           last = 前缀扫描: i 以下最近的非 999 位置
    carry = v // 1000                         carry_in[i] = gen[last[i-1]] # 链头是否点火，一步查到
                                              return (s + carry_in) % 1000

CuPy 里用 cumsum + 一次花式索引实现这个前缀扫描；CUDA C++ 里用教科书式的Kogge–Stone (generate, propagate) 扫描（log₂n ≈ 21 趟布尔操作，顺带把所有 host 同步都省了）。修复后：

CuPy 版（100 万位） 修复前 修复后 加速
端到端 86.7 s 16.7 s 5.2×
其中：最后一次除法 71.4 s 0.9 s 79×
教训：向量化的「平均情况」性能可能藏着 O(n) 的最坏情况，而数值算法（牛顿迭代）恰恰会系统性地制造最坏情况输入。纯整数的级数求和很少产生 Bg 附近的数，多趟传播够用；而 Gauss–Legendre 的牛顿迭代每一步都在制造这种数——快算法把压力转移给了基础设施。

11. 实现 1：CuPy —— 数组级的 GPU 大数库

实现 1 分成两层：主程序 gauss_legendre.py 承载算法（AGM 主循环、牛顿爬升、终止判据），它只调用大数库暴露的原语（add / sub / mul / half / shl / shr / cmp / isqrt…）；大数库 bignum_cupy.py 把每个原语写成对整条 limb 数组的运算（第 4–6 节的全部代码就取自它）——数组常驻显存，每个数组操作由 CuPy 发射成 CUDA kernel，FFT 则走 NVIDIA 的 cuFFT 库。算法层完全不知道自己跑在 GPU 上，这也为第 12 节的 CUDA C++ 版立好了「行为规范」：同一串大数运算（791 次 FFT 乘法、273 步牛顿），逐位一致的输出互为对拍。 

while True:                                    # Gauss–Legendre 主循环（gauss_legendre.py，节选）
    c = bg.cmp(A, Bv)
    diff = bg.sub(A, Bv) if c > 0 else bg.sub(Bv, A)
    agree = 3 * (Lp - len(strip(diff)))        # a、b 已一致的十进制位数
    if 2 * agree >= p + 30:                    # 一致位数再翻一倍就超过 p → 可以出结果了
        break
    A2 = bg.half(bg.add(A, Bv))                # a' = (a+b)/2
    d  = bg.sub(A, A2)                         # a − a'（高位全零，自动变短）
    T  = bg.sub(T, bg.mul_scalar(bg.shr_limbs(bg.mul(d, d), Lp), pn))
    K  = bg.mul(A, Bv)                         # a·b
    Bv, y = rsqrt_ramp(bg, K, warm=y, warm_digits=agree - 9)   # b' = sqrt(ab)，热启动
    A = A2; pn *= 2

S  = bg.add(A, Bv)                             # 收尾：π·10^p = (a+b)²/(4t)
PI = bg.shr_limbs(bg.mul(bg.mul(S, S), recip_ramp(bg, bg.mul_scalar(T, 4), g)), g)

12. 实现 2：自己搭一个百万点 FFT

12.1 规则与动机

实现 1 里 GPU 的 FFT 由 cuFFT 完成——那是 NVIDIA 的闭源库，一个函数调用进去就是黑盒。实现 2 给自己立的规矩是：不用任何主机端数学库（不用 cuFFT、不用 cuBLAS），只允许两个 header-only 的「设备端积木」：cuFFTDx 和 cuBLASDx（NVIDIA MathDx 25.6）。它们的定位很特别：不是「替你做完一个 FFT」，而是给你的 CUDA kernel 里嵌入一段「单个线程块内的小 FFT / 小矩阵乘」，数据必须已经在你自己管理的共享内存或寄存器里。换句话说：百万点的大 FFT 得自己拆、自己拼。

给没写过 CUDA 的读者的最小背景：GPU 程序由成千上万个线程执行，线程按线程块（这里是 256 线程）分组；同一块内的线程可以通过一小块超快的共享内存（这里 80KB）协作，块与块之间只能通过显存见面。cuFFTDx 能做的最大 fp64 FFT 就是「一个块的共享内存装得下」的规模（几千点），而我们需要 220~222 点。

12.2 四步法：大 FFT = 转置 + 两轮小 FFT

把长度 N = P·Q 的数组看成 P×Q 的矩阵（n = Q·i + j），DFT 的求和可以精确地拆成：

X [ k i + P k j ] = ∑ j W Q j k j ⋅ W N j k i ⋅ ( ∑ i x [ Q i + j ] W P i k i )

读法：先沿 i 方向做 Q 个 P 点 FFT（内层求和），乘上「跨行相位」WNj·k_i（twiddle 因子），再沿 j 方向做 P 个 Q 点 FFT（外层求和）。中间夹两次矩阵转置，让每轮小 FFT 的数据在内存里连续：

一个省了大功夫的细节：kernel B 输出的频率顺序天然是「置乱」的（数位反转的混合序）。教科书做法是再加一趟重排恢复自然序，但我们的 FFT 只服务于卷积——正变换后马上点乘、马上逆变换。只要正逆变换使用同一套置乱序，点乘逐元素对齐即可，顺序根本无所谓。于是重排一趟都不用做，逆变换就是正变换的严格镜像。

12.3 kernel B：DFT 就是矩阵乘 —— cuBLASDx 的用武之地

每行 2048 点的 FFT 在一个线程块内完成，而 2048 = 32 × 64 还可以再拆一层。看 DFT 的定义：32 点 DFT 就是「32×32 的常数矩阵 F₃₂（元素 W₃₂g·k）乘一个向量」。把一行数据重排成 32×64 的矩阵后，「对 64 列各做一次 32 点 DFT」恰好是一个复数矩阵乘 C = F₃₂ · X ——这正是 cuBLASDx 的地盘，而且 H100 的 FP64 tensor core（矩阵乘专用硬件单元）会自动接管。剩下的 64 点 FFT 交给 cuFFTDx。同一份数学，两种视角：「FFT 是稀疏分解的矩阵乘，矩阵乘是展开了的 DFT」。

using FFT64 = decltype(cufftdx::Block() + cufftdx::Size<64>() + cufftdx::Type<fft_type::c2c>()
            + cufftdx::Precision<double>() + cufftdx::ElementsPerThread<8>()
            + cufftdx::FFTsPerBlock<32>() + cufftdx::SM<900>() + ...);   // 32 批 64 点 FFT/块

using GEMM32 = decltype(cublasdx::Size<32, 64, 32>()                     // C(32×64) = A(32×32)·B(32×64)
             + cublasdx::Precision<double>() + cublasdx::Type<type::complex>()
             + cublasdx::Arrangement<row_major, row_major, row_major>()
             + cublasdx::BlockDim<FFT64::block_dim.x, FFT64::block_dim.y, 1>()  // 与 FFT 同块形!
             + cublasdx::SM<900>() + cublasdx::Block());
template<bool INV> __global__ void k_stageB(cplx* data, const cplx* fmat, long N) {
    extern __shared__ char smem[];
    dxcplx* row = ... + blockIdx.x * 2048;                     // 本块负责第 j 行
    auto [sa, sb, sc] = cublasdx::slice_shared_memory<GEMM32>(smem);
    // ① G 级：radix-32 DFT 写成矩阵乘 —— C[k_g][s] = Σ_g F32[k_g][g] · row[s+64g]
    cublasdx::copy<GEMM32, ...>(F32_global, a_s);              // DFT 矩阵入共享内存
    cublasdx::copy<GEMM32, ...>(row_global, b_s);              // 行数据入共享内存
    GEMM32().execute(dxcplx{1,0}, a_s, b_s, dxcplx{0,0}, c_s); // FP64 tensor core
    __syncthreads();
    // ② 内 twiddle + ③ S 级：32 批 64 点 FFT（寄存器 API）
    for (i...) rd[i] = cmul(c_s(kg, s), twiddle(-1, s * kg, 1.0/1024));
    __syncthreads();
    FFT64().execute(rd, smem);
    // ④ 外 twiddle（四步法的跨行相位）+ 按存储序写回：pos = 64·k_g + k_s
    for (i...) row[kg*64 + ks] = cmul(rd[i], twiddle(-1, j * (kg + 32L*ks), 2.0/N));
}

精度上有一个必须较真的细节：twiddle 因子 e−2πi·u/D 的辐角。我们所有的 D 都是 2 的幂，u/D 在二进制浮点里精确可表示，配合 CUDA 的 sincospi（自带精确的 π 参数归约）可以做到每个 twiddle ≤1 ulp 误差——不查表、不累积、现场算： 

// e^{±2πi·u/D}，D 是 2 的幂 → u/D 在二进制里是精确值，sincospi 误差 ≤1 ulp
__device__ dxcplx twiddle(double sign2, long num, double inv_den) {
    double s, c;
    sincospi(sign2 * (double)num * inv_den, &s, &c);
    return dxcplx{c, s};
}

32 点 DFT 用「朴素矩阵乘」代替 5 层蝶形，每个输出多累加约 32 项浮点和，误差相当于多两三层蝶形——实测整个 100 万位计算的 FFT 最大舍入误差从纯 cuFFTDx 路径的 1.2×10⁻⁴ 升到 2.1×10⁻⁴，仍比 0.25 的红线低三个数量级。

12.4 其余零件

FFT 之外全是常规 CUDA 工程：装载（limb→复数）、点乘（顺带除 N）、取整+误差归约（0.25 红线每次乘法必查）、Kogge–Stone 进位前瞻（第 10 节）、转置（32×33 共享内存瓦片）、schoolbook 小乘法（两数乘积 ≤ 1600 万时直接平方级算法反而更快）、以及一个一次 cudaMalloc 的 8GB 显存池（指针递增分配 + 作用域回退，峰值实际只用了 403MB）。牛顿迭代的主机端控制流与 Python 版逐行同构。整个实现约 1400 行 CUDA C++。

12.5 自检梯子

手搭 FFT 最怕「悄悄算错」。上线前的分级自检，每级只暴露一类新风险：①加/减/进位 ↔ __int128 对拍；②schoolbook（精确）↔ FFT 乘法交叉对拍；③恒等式 (X+Y)² = X²+2XY+Y² 在全部十种 FFT 规格（Q=4…2048，N=2¹³…2²²）上成立；④牛顿开方/倒数结果与精确值差 ≤3 个最末位；⑤300 → 30 万位 π 与基准逐位比对；⑥最后才是 100 万位。全部一次通过（这要感谢先有 CuPy 实现当「行为规范」——语义逐函数照抄，风险只剩 FFT 本身）。

13.3 cuBLASDx 的 GEMM 路径值不值？

kernel B 有一个编译开关可以把「32 点 DFT=矩阵乘」换回纯 cuFFTDx 的 2048 点 FFT，两版 A/B 对比：端到端 0.882s vs 0.856s（GEMM 路径慢 3%），误差 2.1×10⁻⁴ vs 1.2×10⁻⁴。结论很有教益：在这个访存主导的流水线里，tensor core 的算力几乎是免费的（GEMM 级只占 kernel 时间的零头），但也换不来加速——瓶颈在显存带宽和转置，不在乘加。GEMM 路径的价值在于展示两个 Dx 库如何在同一个 kernel、同一块共享内存里协作，以及「换一种数学视角实现同一变换」的工程可行性。

15. 结语

这次旅程从一个数学定理出发（算术几何平均的二次收敛），落脚在一个线程块的共享内存里（32×64 的矩阵乘与 64 点 FFT）。几件值得带走的事：

• 算法先于实现：二次收敛把「100 万位」压缩成 19 轮大运算——选对算法，问题的形状就先对了一半。
• GPU 快的前提是「大块、规整、可并行」：CuPy 版牛顿爬升里几百次小 kernel 加同步就能吃掉 16 秒，而同样的数学在 CUDA C++ 里整理成大块流水后只要 0.9 秒——上万个核心要用「整片数组」去喂。
• 快算法会把压力转移给基础设施：每轮一次全精度开方逼出了精度爬升与热启动；牛顿迭代的特殊数值形态逼出了进位前瞻。性能问题往往藏在「平均情况很好」的原语的最坏情况里。
• GPU 上「快」有两种：浮点吞吐与调度开销。百万位这个规模恰好在两者的分界线上——cuFFT 的浮点我们打不过，但把几百次同步压成零之后，端到端还是快了 19 倍。
• DFT 与 GEMM 本是一家：cuFFTDx 和 cuBLASDx 在同一个 kernel 里各拆一半 2048 点 FFT，tensor core 顺手把 DFT 矩阵乘掉——这是「库函数之下」那一层世界的日常。
